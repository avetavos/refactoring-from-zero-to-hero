Move Statements into Function

Intent

หัวข้อที่มีชื่อว่า “Intent”

เมื่อโค้ดบรรทัดหนึ่งจับคู่กับการเรียกฟังก์ชันบางตัวเสมอ — รันก่อนหรือหลังมันทันที ที่ทุก ๆ จุดเรียก — บรรทัดนั้นควรอยู่ ภายใน ฟังก์ชัน ย้ายมันเข้าไป ตอนนี้ฟังก์ชันเป็นเจ้าของงานทั้งหมดอย่างครบถ้วน และผู้เรียกก็หดเหลือเพียงการเรียกครั้งเดียวที่ทำพลาดไม่ได้

The smell

หัวข้อที่มีชื่อว่า “The smell”

คุณพบคำสั่งเตรียมการหรือคำสั่งปิดท้ายอันเดิมถูก copy-paste ไว้เหนือหรือใต้การเรียกฟังก์ชันทุกที่ที่มันปรากฏ บางทีผู้เรียก renderName ทุกตัวเขียน opening tag ก่อน หรือผู้เรียก fetchOrder ทุกตัว log บรรทัดเดียวกันหลังจากนั้น คำสั่งที่ทำซ้ำนั้นเป็นส่วนหนึ่งของพฤติกรรมที่ผู้เรียกคาดหวัง แต่กลับอยู่นอกฟังก์ชัน ที่ซึ่งผู้เรียกรายใหม่ทุกรายต้องจำให้เพิ่มมันเข้าไป ลืมสักครั้งเดียวก็ได้บั๊กเงียบ ๆ การทำซ้ำนี้เป็นสัญญาณว่าคำสั่งนั้นอยากจะย้ายเข้าไป

Before → After

หัวข้อที่มีชื่อว่า “Before → After”

ฟังก์ชันที่ปล่อย HTML รูปภาพของบุคคล ผู้เรียกทุกตัวเขียนบรรทัด heading <p> อันเดิมก่อนจะเรียกมันทันที heading นั้นจริง ๆ แล้วเป็นส่วนหนึ่งของ “render บล็อกรูปภาพ” เราจึงย้ายมันเข้าไป

  • Python
  • Go

Mechanics

หัวข้อที่มีชื่อว่า “Mechanics”
  1. ยืนยันว่าคำสั่งที่เป็นตัวเลือกนั้นรันติดกับการเรียกที่ ทุก จุด ถ้าแม้เพียงผู้เรียกรายเดียวทำต่างออกไป การย้ายนี้ผิด — ความแตกต่างนั้นเป็นของจริง
  2. ถ้าจุดเรียกมีโค้ดอื่นอยู่ ให้ใช้ Extract Function ก่อนเพื่อแยกเฉพาะการเรียกบวกกับคำสั่งที่ทำซ้ำออกเป็น helper เดียว ตอนนี้คุณมีที่ทำงานเพียงที่เดียว
  3. ย้ายคำสั่งที่ทำซ้ำเข้าไปในฟังก์ชันเป้าหมาย — ไว้บนสุดถ้ามันรันก่อนการเรียก ไว้ล่างสุดถ้ามันรันหลังการเรียก
  4. รันเทสต์ของคุณ
  5. ลบคำสั่งที่ตอนนี้ซ้ำซ้อนออกจากผู้เรียกแต่ละราย ทีละราย โดยรันเทสต์หลังการลบแต่ละครั้ง
  6. ถ้าคุณสร้าง helper ชั่วคราวในขั้นที่ 2 ให้ inline มันเมื่อผู้เรียกทุกรายสะอาดแล้ว

When to use / trade-offs

หัวข้อที่มีชื่อว่า “When to use / trade-offs”

ใช้สิ่งนี้เมื่อคำสั่งหนึ่งเป็นส่วนหนึ่งของความรับผิดชอบของฟังก์ชันอย่างแท้จริง แต่บังเอิญอยู่นอกมัน การพับมันเข้าไปจะกำจัดการทำซ้ำและทำให้ฟังก์ชันเป็นหน่วยที่ครบถ้วนและใช้ผิดได้ยาก

อย่าใช้มันเมื่อผู้เรียกแตกต่างกันในเรื่องว่าจะรันคำสั่งหรือไม่ หรือรันอย่างไร — การบังคับขั้นตอนร่วมลงบนฟังก์ชันที่ผู้เรียกบางรายอยากข้าม เป็นเพียงการแลกการทำซ้ำไปกับ flag parameter ซึ่งแย่กว่า การ refactor ที่ตรงข้ามคือ Move Statements to Callers: เมื่อฟังก์ชันมัดขั้นตอนที่มีเพียงผู้เรียก บางราย ต้องการ ให้ดันมันออกไป

🧹 ย้ายเฉพาะสิ่งที่แชร์กันจริง ๆ

การทดสอบนั้นง่าย: ผู้เรียก ทุกราย รันคำสั่งนี้เหมือนกันเป๊ะ ติดกับการเรียกหรือไม่? ถ้าใช่ ให้ย้ายมันเข้าไป ถ้าคุณต้องเพิ่ม boolean เพื่อรองรับข้อยกเว้น ให้หยุด — คำสั่งนั้นไม่ได้แชร์กัน และมันควรอยู่ที่ผู้เรียก

เมื่อใดที่คำสั่งหนึ่งเป็นตัวเลือกที่ดีในการย้ายเข้าไปในฟังก์ชัน?
