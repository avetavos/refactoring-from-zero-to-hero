## Intent

The **simplifying conditionals** family is about decision logic: the branches, guards, and switches that decide *what the program does next*. A condition that started as a single clear question grows extra clauses, nests inside other conditions, and sprouts copies across the codebase. These refactorings untangle that growth so the reader can see, at a glance, which case leads where.

## The smell

Complex conditional logic is one of the loudest signals that a function is hard to maintain. The warning signs are familiar: a boolean expression so long you have to read it twice, an arrow of nested `if` blocks marching off the right edge of the screen, a `switch` on a "type" field repeated in five functions, and `if (x == null)` scattered through every method that touches `x`. Each of these has a named cure, and each cure leaves the behaviour identical while making the intent obvious.

## What this module covers

Eight refactorings, each small and reversible:

- **[Decompose Conditional](/refactoring-from-zero-to-hero/en/simplifying-conditionals/decompose-conditional/)** — extract the test, the then-branch, and the else-branch into well-named functions so the structure reads like a sentence: *if it is a peak day, charge the peak rate, otherwise the regular rate.*
- **[Consolidate Conditional Expression](/refactoring-from-zero-to-hero/en/simplifying-conditionals/consolidate-conditional/)** — when several separate checks all lead to the same result, combine them into one condition and name it, so the reader learns *why* rather than *how many ways*.
- **[Replace Nested Conditional with Guard Clauses](/refactoring-from-zero-to-hero/en/simplifying-conditionals/replace-nested-conditional-with-guard-clauses/)** — flatten deep `if/else` by returning early for the special cases, leaving the main path unindented and obvious.
- **[Replace Conditional with Polymorphism](/refactoring-from-zero-to-hero/en/simplifying-conditionals/replace-conditional-with-polymorphism/)** — when a `switch` or `if` on a type recurs, push each branch into a subclass method so the language dispatches for you.
- **[Introduce Special Case](/refactoring-from-zero-to-hero/en/simplifying-conditionals/introduce-special-case/)** — replace a repeated null-or-special check with an object that responds with sensible defaults, so callers stop asking "is it missing?".
- **[Introduce Assertion](/refactoring-from-zero-to-hero/en/simplifying-conditionals/introduce-assertion/)** — make an assumption a section of code silently relies on explicit, so a violated assumption fails loudly at its source instead of corrupting data downstream.
- **[Replace Control Flag with Break/Return](/refactoring-from-zero-to-hero/en/simplifying-conditionals/replace-control-flag-with-break/)** — delete a boolean that only exists to steer a loop and let `break`, `return`, or `continue` express the exit where it actually happens.
- **[Replace Error Code with Exception](/refactoring-from-zero-to-hero/en/simplifying-conditionals/replace-error-code-with-exception/)** — stop reporting failure with an ignorable sentinel; raise an exception, or in Go and Rust return an explicit `error`/`Result`, so failure cannot be mistaken for success.

Each lesson shows the same before-and-after in TypeScript, Python, Go, and Rust, then walks through the safe, ordered mechanics — running your tests between every step.

<Callout title="Make the decision visible, not the plumbing">
The goal is never to remove conditionals for their own sake. A program has to make decisions. The goal is to make each decision *legible*: one question per condition, a name for every non-obvious test, and the common path easy to find. If a refactoring here makes the logic harder to read, you have gone too far — stop and reverse it.
</Callout>
