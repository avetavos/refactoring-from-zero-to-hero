## Intent

Take a run of separate conditionals that each guard the *same* outcome and fold them into one combined condition, then extract that condition into a named function. Three scattered checks with the same `return 0` become a single `if (isNotEligible(employee)) return 0`, and the reason for the result is captured in the name.

## The smell

This is the cure for a **sequence of conditionals with identical results**. Each check on its own looks innocent, but together they hide a single idea behind three lines of plumbing. The reader has to notice that all three produce the same value before understanding that they are really one question — *is this person ineligible?* Consolidating makes that one question explicit, and a named helper tells the next reader *why* without making them reverse-engineer it.

## Before → After

A disability-benefit amount returns zero for several disqualifying conditions. Before, each disqualifier is its own `if`. After, they are one named check.

## Mechanics

1. Confirm that the conditionals truly share a result and have no side effects between them. If one check mutates state the next relies on, do not merge.
2. Combine the tests with `||` when they all lead to the same branch (or `&&` when the structure is nested checks that must all pass). Keep the combined condition next to the shared result.
3. **Run your tests.** The behaviour must match the chain of separate `if`s exactly, including short-circuit order.
4. Apply **Extract Function** to the combined condition, naming it after the question it answers (`isNotEligible`).
5. Run your tests again.

## When to use / trade-offs

Reach for Consolidate Conditional Expression whenever you see consecutive checks that funnel into the same outcome. It is most valuable as a setup move: once the checks are one named function, you can reuse that function, and it often clears the way for **Replace Nested Conditional with Guard Clauses**.

Do not consolidate when the checks are genuinely independent decisions that happen to share a value today but may diverge tomorrow — merging them would couple unrelated rules. And never merge across a side effect; the combined expression's short-circuiting could skip work the original chain performed.

<Callout title="Consolidate only when the result is genuinely the same">
The trigger is identical outcomes, not similar-looking conditions. If two checks return different values, leave them apart — forcing them together hides a real difference. Merge for clarity, never to cut line count.
</Callout>
