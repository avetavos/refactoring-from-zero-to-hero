## Intent

บาง field เป็นส่วนหนึ่งของ identity ของอ็อบเจกต์: เลขบัญชี, timestamp การสร้าง, สกุลเงิน พวกมันควรถูกเลือกตอนอ็อบเจกต์เกิดและไม่ถูกแตะอีกเลย ถ้า field แบบนั้นมี setter ใครก็ตามสามารถ mutate มันเงียบ ๆ ภายหลังได้ ย้ายการกำหนดค่าเข้าไปใน constructor และ **ลบ setter ทิ้ง** เพื่อให้ field ถูกตั้งเพียงครั้งเดียวและอ็อบเจกต์ไม่เปลี่ยนแปลงในแง่นั้น

## The smell

คุณสร้างอ็อบเจกต์ แล้วเรียก setter ทันทีเพื่อให้ค่าที่จะไม่มีวันเปลี่ยนอย่างถูกต้อง — `new Account(); account.setId(42)` การเกิดสองขั้นชวนให้บุคคลที่สามเรียก setter อีกครั้ง ห่างไกลจากการสร้าง และทำลาย identity ของอ็อบเจกต์ setter ที่มีอยู่เพียงเพื่อให้ถูกเรียกครั้งเดียวตอนเริ่มคือ setter ที่ไม่ควรมีอยู่

## Before → After

`Account` เปิดเผย `id` ที่ตั้งค่าได้ เรารับ id ใน constructor และลบ `setId` ทำให้ identity ถาวร ใน Go และ Rust ไม่มี setter ตั้งแต่แรก — ประเด็นคือกำหนดค่าตอนสร้างและไม่เปิดเผยตัว mutate

## Mechanics

1. ถ้า constructor ยังรับ field ไม่ได้ ให้เพิ่มมันเป็น parameter และกำหนดค่ามันข้างใน constructor
2. อัปเดตผู้เรียกทุกรายที่สร้างอ็อบเจกต์ให้ส่งค่าผ่าน constructor แทนที่จะผ่าน setter **รันเทสต์ของคุณ** หลังแต่ละราย
3. ค้นหาการเรียก setter ที่ยังเหลืออยู่ ถ้าผู้เรียกยัง mutate field หลังการสร้าง ให้ตัดสินว่านั่นเป็นความต้องการจริงไหม — ถ้าใช่ field นี้ไม่ใช่ตัวเลือก; ถ้าไม่ ให้นำมันผ่าน constructor
4. เมื่อไม่มีผู้เรียกใช้ setter แล้ว ก็ลบมันทิ้ง
5. ที่ภาษาอนุญาต ให้ทำเครื่องหมาย field เป็น `readonly`/`final`/private-ที่ไม่มีตัว mutate เพื่อให้ compiler บังคับความไม่เปลี่ยนแปลง

## When to use / trade-offs

ใช้สิ่งนี้กับ field ที่นิยาม identity หรือตายตัวตลอดอายุของอ็อบเจกต์ และกับ value object ที่คุณต้องการปฏิบัติว่าไม่เปลี่ยนแปลง เพื่อให้แชร์และ cache ได้อย่างอิสระ การลบ setter หดพื้นที่ผิวสำหรับ bug: ค่าที่เปลี่ยนไม่ได้ก็เปลี่ยนในเวลาที่ผิดไม่ได้

ข้อแลกเปลี่ยนคือแรงกดดันที่ constructor ถ้ามี field แบบนี้หลายตัว รายการ parameter ของ constructor อาจยาวขึ้น — นั่นเป็นสัญญาณให้ทำ **Introduce Parameter Object** หรือ builder ไม่ใช่เหตุผลที่จะคง setter ไว้ และ state ที่เปลี่ยนได้จริง ๆ — ยอดเงิน, สถานะ — ยังต้องมีวิธีเปลี่ยนที่ควบคุมได้; อย่าถอด setter ที่จำลองการเปลี่ยนแปลงจริงที่ดำเนินอยู่ เฟรมเวิร์กที่ต้องการ constructor ไม่มี argument และ field injection ก็อาจบังคับให้ต้องประนีประนอมตรงนี้
