## Intent

ถ้าฟังก์ชันสามารถคำนวณค่าของ parameter หนึ่งจากข้อมูลที่มันรับมาอยู่แล้ว parameter นั้นก็เป็นน้ำหนักที่ตายแล้ว ผู้เรียกทุกคนต้องคำนวณและส่งมัน และพวกเขาขัดแย้งกันได้ ลบมันออก แล้วให้ฟังก์ชันถามหาค่านั้นเอง — **query** — เพื่อให้มีแหล่งความจริงเดียว และมีสิ่งให้ทำผิดที่ call site น้อยลงหนึ่งอย่าง

## The smell

parameter หนึ่งถูกส่งในแบบเดียวกันเสมอ คำนวณจาก argument อีกตัวที่ฟังก์ชันมีอยู่แล้ว คุณเห็นผู้เรียกรันการคำนวณเล็ก ๆ แบบเดียวกันก่อนทุกการเรียก แล้วยื่นผลลัพธ์เข้าไป แย่กว่านั้น ผู้เรียกสองรายคำนวณมันต่างกันเล็กน้อย ฟังก์ชันจึงทำงานไม่สม่ำเสมอขึ้นกับว่าใครเรียก parameter เสนอความยืดหยุ่นที่ไม่มีใครต้องการจริง ๆ

## Before → After

`finalPrice` รับ `quantity` และ `discount` แต่ discount ถูกกำหนดทั้งหมดโดย quantity ผู้เรียกคอยคำนวณมัน เราลบ parameter ออกและคำนวณ discount ข้างใน

## Mechanics

1. ยืนยันว่า parameter คำนวณได้ทั้งหมดจากข้อมูลที่ฟังก์ชันมีอยู่แล้ว — parameter อื่น, field หรือ query ที่เข้าถึงได้ ถ้ามันขึ้นกับอะไรภายนอก ให้หยุด: การลบมันจะลักลอบนำ dependency ที่ซ่อนอยู่เข้ามา
2. แยกการคำนวณออกเป็น query ถ้ามันยังไม่ใช่
3. ข้างในฟังก์ชัน แทนที่การใช้ parameter แต่ละจุดด้วยการเรียก query นั้น
4. ลบ parameter ออกจาก signature แล้วลบการคำนวณที่ตอนนี้ตายแล้วออกจากผู้เรียกทุกราย **รันเทสต์ของคุณ** หลังแต่ละผู้เรียก

## When to use / trade-offs

ใช้สิ่งนี้เมื่อผู้เรียกคำนวณ argument ในแบบเดียวกันทั้งหมด, เมื่อค่าถูกกำหนดทั้งหมดโดยข้อมูลที่ฟังก์ชันเข้าถึงได้ หรือเมื่อคุณต้องการลบความเสี่ยงที่ผู้เรียกจะส่งค่าที่ไม่สอดคล้องกัน parameter ที่น้อยลงหมายถึง interface ที่เรียบง่ายและใช้ผิดได้ยากขึ้น

ทางกลับคือ **Replace Query with Parameter** บางครั้งการคำนวณค่า *ข้างใน* ฟังก์ชันสร้าง dependency ที่ซ่อนอยู่ — ฟังก์ชันเอื้อมไปหา global state, นาฬิกา หรือ singleton — ทำให้ทดสอบและใช้เหตุผลกับมันยากขึ้น ที่นั่นคุณทำตรงกันข้าม: ยกการคำนวณออกมา รับค่าเป็น parameter และให้ผู้เรียกจัดหามัน นั่นแลก signature ที่เพรียวกับ dependency ที่ชัดเจนและฉีดเข้าได้ เลือกจากสิ่งที่คุณให้คุณค่ามากกว่าตรงนี้: interface ที่เล็ก หรือฟังก์ชันที่ไม่มีการเอื้อมที่ซ่อนอยู่
