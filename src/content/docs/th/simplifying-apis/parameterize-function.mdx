## Intent

มองหา function สองตัวขึ้นไปที่เนื้อในเหมือนกันยกเว้นค่าที่ฝังไว้ตายตัวค่าเดียว แล้วยุบมันให้เหลือ function เดียวที่รับค่านั้นเป็นพารามิเตอร์ ตอนนี้ตรรกะที่ซ้ำซ้อนอยู่ในที่เดียวเป๊ะ ๆ และความแตกต่างระหว่างตัวแปรเดิมกลายเป็นอาร์กิวเมนต์ที่มองเห็นได้ที่จุดเรียกใช้

## Code Smell

นี่คือรสชาติหนึ่งของ **Duplicated Code** คุณเห็น `discountFivePercent` และ `discountTenPercent` นั่งอยู่ข้าง ๆ กัน เหมือนกันทุกอย่างยกเว้นตัวเลข หรือ `tierBronze` และ `tierSilver` ที่ต่างกันแค่ค่า threshold ทุกครั้งที่ตรรกะที่ใช้ร่วมกันต้องได้รับการแก้ คุณต้องจำให้ได้ว่าต้องไปแก้ทุก clone — และวันหนึ่งคุณก็จะลืมไปสักตัว ค่าคงที่ที่ฝังไว้ตายตัวคือสิ่งเดียวที่แปรเปลี่ยน มันจึงอยากเป็นอาร์กิวเมนต์ ไม่ใช่สำเนาของทั้ง function

## Before → After

function สองตัวที่คำนวณราคาหลังหักส่วนลด ต่างกันแค่อัตรา หลังจากนั้น function เดียวรับอัตราเป็นพารามิเตอร์

## Mechanics

1. เลือก function ที่เกือบเหมือนกันตัวหนึ่งมาเป็นแม่แบบ
2. เพิ่มพารามิเตอร์สำหรับค่าที่แปรเปลี่ยน ตั้งชื่อให้สื่อความหมายอย่าง `rate` หรือ `threshold` ไม่ใช่ `value`
3. ภายในเนื้อใน ให้แทน literal ที่ฝังไว้ตายตัวด้วยพารามิเตอร์ใหม่ **รันเทสต์ของคุณ** — behavior ควรไม่เปลี่ยนสำหรับกรณีเดิมของ function ตัวนั้น เมื่อคุณส่งค่าคงที่เดิมของมันเข้าไป
4. สำหรับ clone ที่เหลือแต่ละตัว ให้เปลี่ยนเส้นทางผู้เรียกใช้ของมันไปยัง function ที่ถูก parameterize แล้ว โดยส่ง literal เฉพาะของ clone นั้นเข้าไปเป็นอาร์กิวเมนต์
5. **รันเทสต์ของคุณ** หลังเปลี่ยนเส้นทางแต่ละ clone เพื่อให้ความล้มเหลวชี้ไปยังการเรียกใช้จุดเดียว
6. เมื่อไม่มีอะไรเรียกใช้ clone เดิมแล้ว ให้ลบมันทิ้ง
7. พิจารณาว่าพารามิเตอร์ใหม่ควรถูกตรวจสอบความถูกต้องหรือไม่ — ตัวอย่างเช่น อัตราส่วนลดที่อยู่นอกช่วงศูนย์ถึงหนึ่งน่าจะเป็นบั๊กที่ควรปฏิเสธ

## When to use / trade-offs

หยิบท่านี้มาใช้เมื่อ function ต่างกันแค่ค่าคงที่ เมื่อคุณกำลังจะ copy-paste function แล้วปรับตัวเลขตัวเดียว หรือเมื่อชุดของตัวแปรที่มีชื่อต่าง ๆ เพิ่มขึ้นเรื่อย ๆ การ parameterize ฆ่าความซ้ำซ้อนและทำให้มิติที่แปรเปลี่ยนชัดเจนและค้นพบได้

ข้อแลกเปลี่ยนคืออาร์กิวเมนต์แบบ literal ที่จุดเรียกใช้อาจอธิบายตัวเองได้น้อยกว่า function ที่ตั้งชื่อมาดี — `0.05` เปล่า ๆ บอกอะไรน้อยกว่า `priceWithFivePercentOff` ถ้าชุดของค่ามีจำนวนน้อยและสื่อความหมาย ค่าคงที่ที่มีชื่อหรือ enum ที่จุดเรียกใช้ก็คืนความชัดเจนนั้นกลับมา และจงต้านทานการ parameterize มากเกินไป: function ที่มีปุ่มปรับแปดปุ่มใช้ยากกว่า clone ที่มันมาแทนที่

| ใช้ Parameterize Function เมื่อ | หลีกเลี่ยงเมื่อ |
|---|---|
| function สองตัวต่างกันแค่ literal เดียว | ความแตกต่างคือ behavior ไม่ใช่ data |
| มีตัวแปรใน namespace เพิ่มขึ้นเรื่อย ๆ | parameter ใหม่ทำให้ signature อ่านยากขึ้น |
| จะ copy-paste function แล้วเปลี่ยนตัวเลขเดียว | function ที่ได้มี parameter เกิน 4 ตัว |

> ⚠️ ไม่ควร Parameterize Function เมื่อ:
> - ความแตกต่างระหว่างสอง function คือ behavior — ให้ Remove Flag Argument แทน
> - ค่าที่จะเป็น parameter มีชุดจำกัดและมีความหมาย — ใช้ enum หรือแยก function แทน
> - เพิ่ม parameter เพราะ "อาจจะต้องการ" ในอนาคต — YAGNI

<Callout title="Parameterize ที่ค่า ไม่ใช่ที่ behavior">
ท่านี้เปล่งประกายเมื่อความแตกต่างเพียงอย่างเดียวคือ *ข้อมูล* — อัตรา ขีดจำกัด ป้ายกำกับ เมื่อความแตกต่างคือ *behavior* — ตัวแปรหนึ่งบันทึก log และอีกตัวส่งอีเมล — พารามิเตอร์ boolean ตัวเดียวมักจะแตกเนื้อในออกเป็นสอง ซึ่งคือกลิ่นของ Flag Argument ให้พารามิเตอร์บรรยายค่า และให้การแยก function บรรยาย behavior
</Callout>
