## Intent

นำ function ที่ใช้อาร์กิวเมนต์แบบ boolean เพื่อเลือกระหว่างสอง behavior ที่แตกต่างกัน แล้วแทนที่ด้วยสอง function แยกกัน — หนึ่ง function ต่อหนึ่ง behavior — แต่ละตัวมีชื่อที่บอกว่ามันทำอะไร ผู้เรียกใช้เลิกส่ง `true` ที่เข้ารหัสยาก แล้วเริ่มเรียก function ที่ชื่อของมันคือคำตอบ

## Code Smell

flag argument ซ่อนเจตนาไว้ที่จุดเรียกใช้ เมื่อคุณอ่าน `sendEmail(order, true)` ตัว `true` ไม่ได้บอกอะไรคุณเลย — คุณต้องเปิด function เพื่อจะรู้ว่า `true` หมายถึง "ส่งด่วน" ในขณะที่ `false` หมายถึง "เข้าคิวไว้ทีหลัง" boolean กำลังทำงานที่ชื่อ function ควรทำ มันยังมักทำให้เนื้อในแตกแขนงด้วย: `if (urgent) ... else ...` ที่ยัดสองกระแสงานที่ไม่เกี่ยวข้องกันเข้ามาใน function เดียว ทำให้ทั้งสองอ่านไม่สะอาดและทั้งคู่ต้องถูกเทสต์ผ่านประตูเดียวกัน

## Before → After

function แจ้งเตือนที่ boolean เลือกระหว่างเส้นทางด่วนและเส้นทางปกติ หลังจากนั้น สอง function ที่มีชื่อ แต่ละตัวมุ่งเน้น

## Mechanics

1. ยืนยันว่า flag เลือกระหว่าง *behavior* จริง ๆ ไม่ใช่แค่ค่า ถ้ามันแค่สลับตัวเลขหรือป้ายกำกับ Parameterize Function อาจเหมาะกว่าการแยก
2. สร้าง function ใหม่สำหรับแขนงหนึ่งของ flag ตั้งชื่อให้จับเจตนาของแขนงนั้น อย่าง `notifyUrgent`
3. คัดลอกตรรกะของแขนงที่เกี่ยวข้องเข้าไปใน function ใหม่ แล้วทิ้งแขนงที่ตายแล้ว **รันเทสต์ของคุณ**
4. หาผู้เรียกใช้ทุกตัวที่ส่ง flag ด้วยค่านั้น แล้วเปลี่ยนเส้นทางไปยัง function ใหม่ พร้อมลบอาร์กิวเมนต์ boolean ออก
5. ทำซ้ำขั้นที่ 2 ถึง 4 สำหรับอีกแขนงหนึ่ง
6. **รันเทสต์ของคุณ** หลังการเปลี่ยนเส้นทางแต่ละครั้ง เพื่อให้ความล้มเหลวชี้ไปยังผู้เรียกใช้จุดเดียว
7. เมื่อไม่มีผู้เรียกใช้ตัวใดส่ง flag อีกแล้ว ให้ลบ function เดิมทิ้ง

## When to use / trade-offs

แยก flag argument ออกเมื่อใดก็ตามที่พารามิเตอร์ boolean เปลี่ยน *สิ่งที่ function ทำ* เมื่อใดที่ `true` หรือ `false` แบบ literal ที่จุดเรียกใช้อ่านไม่รู้เรื่อง หรือเมื่อใดที่เนื้อในแตกออกเป็นสองกระแสงานที่แทบไม่มีอะไรร่วมกัน สอง function ที่มีชื่อทำให้การเรียกใช้แต่ละครั้งอธิบายตัวเองได้ และให้แต่ละเส้นทางถูกเทสต์ได้โดยตรง

ต้นทุนคือมี function ใน namespace มากขึ้น และ code เตรียมการที่ใช้ร่วมกันจริง ๆ บางส่วนอาจไปอยู่ในสองที่ — ดึงส่วนที่ใช้ร่วมกันนั้นออกมาเป็น helper ส่วนตัวเพื่อไม่ให้คุณทำซ้ำมัน จงใช้สามัญสำนึกด้วย: flag ที่แค่พลิกข้อมูลการตั้งค่า โดยไม่มีการแตกแขนงเชิง behavior คงไว้ได้ไม่เป็นไร กลิ่นที่ว่านี้เจาะจงคือ boolean ที่ *ตัดสินใจเลือกระหว่าง behavior*

| ใช้ Remove Flag Argument เมื่อ | หลีกเลี่ยงเมื่อ |
|---|---|
| boolean เลือกระหว่าง behavior สองแบบ | boolean แค่สลับค่า config ไม่มีการ branch |
| `true`/`false` ที่ call site อ่านไม่รู้เรื่อง | ทั้งสอง path แชร์ logic ส่วนใหญ่ร่วมกัน |
| สองแขนงแทบไม่มี code ร่วมกัน | มี flags มากกว่า 1 ตัวซ้อนกัน — Extract Class แทน |

> ⚠️ ไม่ควร Remove Flag Argument เมื่อ:
> - flag เป็น enum ที่มี 3+ ค่า — แยกแต่ละค่าออกเป็น function อาจสร้าง duplication มากกว่าเดิม
> - สองแขนง share prep logic มาก — extract helper ก่อนค่อยแยก function
> - caller ผ่าน flag จาก variable — การแยก function อาจแค่ย้าย if ไปข้างนอก

<Callout title="ให้ชื่อ function ตอบคำถาม">
flag argument บังคับให้ผู้เรียกใช้เข้ารหัสการตัดสินใจเป็นข้อมูลที่ผู้อ่านต้องถอดรหัส สอง function ย้ายการตัดสินใจนั้นเข้าไปในชื่อ เพื่อให้ `notifyUrgent("Server down")` อ่านได้เหมือนประโยค จุดเรียกใช้ควรบอกคุณว่าเกิดอะไรขึ้น — ไม่ใช่ทำให้คุณต้องไปค้นหาเอง
</Callout>
