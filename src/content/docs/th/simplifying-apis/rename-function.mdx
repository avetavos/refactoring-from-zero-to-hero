## เจตนา

เปลี่ยนชื่อของฟังก์ชันหรือตัวแปรให้มันบอกว่าสิ่งนั้นเป็นอะไรหรือทำอะไรจริง ๆ ชื่อใหม่ควรทำให้ผู้อ่านเข้าใจการเรียกใช้ได้โดยไม่ต้องเปิดดูเนื้อใน จากนั้นอัปเดตผู้เรียกใช้ทุกจุดให้ใช้ชื่อนั้น ไม่มีอะไรเกี่ยวกับพฤติกรรมเปลี่ยน — เปลี่ยนแค่ถ้อยคำ

## กลิ่นไม่ดี

ชื่อที่ชวนเข้าใจผิดแย่กว่าการไม่มีชื่อเสียอีก ฟังก์ชันชื่อ `getCustomer` ที่แอบสร้างลูกค้าขึ้นมาเองหากไม่มีอยู่ ตัวแปรชื่อ `data` ที่เก็บราคาเพียงค่าเดียว flag ชื่อ `check` ที่จริง ๆ แล้ว *ทำการ* ลบทิ้ง — แต่ละอย่างบังคับให้ผู้อ่านต้องเพิกเฉยต่อชื่อแล้วไปศึกษาเนื้อในแทน นั่นคือความล้มเหลวที่ชื่อที่ดีมีไว้เพื่อป้องกัน เมื่อใดที่คุณพบว่าตัวเองกำลังอ่าน implementation เพียงเพื่อจะรู้ว่าสิ่งนั้นชื่ออะไร ให้เปลี่ยนชื่อมัน

## ก่อน → หลัง

ฟังก์ชันที่ชื่อบอกน้อยกว่าสิ่งที่มันทำมาก บวกกับตัวแปรในเครื่องที่คลุมเครือ หลังจากนั้น ชื่อทั้งสองสื่อความหมายของมันออกมา

## กลไก

1. ตัดสินใจเลือกชื่อใหม่ ลองพูดมันออกมาดัง ๆ: มันบรรยายผลลัพธ์ ไม่ใช่กลไกใช่ไหม คนแปลกหน้าจะเข้าใจจุดเรียกใช้ได้โดยไม่ต้องอ่านเนื้อในหรือไม่
2. ถ้าฟังก์ชันเป็นส่วนหนึ่งของ API ที่เผยแพร่แล้ว ให้พิจารณาคงชื่อเดิมไว้เป็น wrapper บาง ๆ ที่ส่งต่อไปยังชื่อใหม่ เพื่อไม่ให้ผู้เรียกใช้เดิมพัง
3. เปลี่ยนชื่อที่ declaration พึ่งฟีเจอร์ rename-symbol ของ editor ได้หากมันเชื่อถือได้ แต่ต้องตรวจสอบขอบเขตที่มันแตะต้อง
4. อัปเดตผู้เรียกใช้ทุกจุด — และคอมเมนต์หรือสตริงทุกตัวที่อ้างถึงชื่อเดิม
5. **รันเทสต์ของคุณ** การเปลี่ยนชื่อไม่ควรเปลี่ยนพฤติกรรม ดังนั้นชุดเทสต์ที่ผ่านยืนยันว่าคุณแค่ขยับถ้อยคำเท่านั้น
6. เมื่อผู้เรียกใช้ทั้งหมดใช้ชื่อใหม่แล้ว ให้ลบ wrapper ชั่วคราวจากขั้นที่ 2 ทิ้ง

## เมื่อไรควรใช้ / ข้อแลกเปลี่ยน

เปลี่ยนชื่อเมื่อใดก็ตามที่ชื่อทำให้คุณต้องหยุดคิด เมื่อใดที่คุณต้องอ่านเนื้อในเพื่อจะรู้ว่าสิ่งนั้นคืออะไร หรือเมื่อใดที่พฤติกรรมของโค้ดเคลื่อนออกห่างจากสิ่งที่ชื่อสัญญาไว้ตอนแรก มันเป็นการรีแฟกเตอร์ที่มีความเสี่ยงต่ำที่สุดเท่าที่มี และเป็นหนึ่งในแบบที่ให้คุณค่าสูงที่สุด เพราะชื่อที่ชัดเจนถูกอ่านบ่อยกว่าถูกเขียนมาก

ต้นทุนคือความวุ่นวายเชิงกลไกที่กระจายไปทั่วผู้เรียกใช้ และสำหรับ API สาธารณะ อาจเป็นการเปลี่ยนแปลงที่ทำให้ของเดิมพัง บรรเทาได้ด้วยเครื่องมือ rename อัตโนมัติ และในจุดที่สัญญามีความเสี่ยง ให้มีช่วงเวลา deprecation ความเสี่ยงแทบจะคุ้มเสมอ: ผู้อ่านทุกคนที่คุณช่วยให้รอดพ้นจากชื่อที่ชวนสับสน ตอบแทนคุณกลับมา
