## เจตนา

ฟังก์ชันบางอันอ่านยากไม่ใช่เพราะมันยาว แต่เพราะ *วิธีการ* นั้นซับซ้อนยุ่งเหยิง — ลูปที่เขียนเองทั้งที่การ lookup ก็ทำได้ โซ่ของเงื่อนไขที่เขียนซ้ำสิ่งที่ standard library มีให้อยู่แล้ว Substitute Algorithm สลับ body ทั้งหมดเป็นอันที่เรียบง่ายกว่าซึ่งให้ผลลัพธ์เหมือนกัน ไม่เหมือนการเคลื่อนไหวแบบ composing ที่เล็กกว่า อันนี้แทนที่ *กลยุทธ์* ไม่ใช่แค่ถ้อยคำ ดังนั้นมันจึงพึ่งพาเทสต์ของคุณอย่างหนักเพื่อพิสูจน์ว่าทั้งสองวิธีให้ผลตรงกัน

## smell

ฟังก์ชันทำงานได้ แต่เมื่ออ่านมันคุณพบว่าตัวเองต้องจำลองมันในหัวเพื่อให้แน่ใจว่าถูกต้อง ตรรกะรู้สึกหนักหน่วงกว่าที่ปัญหาสมควรได้รับ: การค้นหาด้วยมือผ่าน list ทั้งที่การตรวจสอบสมาชิกใน set ก็ตอบได้ หรือลำดับของ `if` แตกแขนงที่สรุปได้ว่า "ค่านี้เป็นหนึ่งในไม่กี่ค่านี้หรือไม่?" เมื่อคุณอธิบายงานของฟังก์ชันได้ในประโยคเรียบ ๆ ประโยคเดียวแต่โค้ดกลับใช้ทั้งย่อหน้า มีอัลกอริทึมที่ชัดเจนกว่ารออยู่แน่นอน

## Before → After

ฟังก์ชันตัดสินใจว่ารหัสประเทศได้รับการรองรับหรือไม่ด้วยการวนลูปและพลิก flag ตัวแทนใช้การตรวจสอบสมาชิกโดยตรงกับ set ที่กำหนดไว้ตายตัว — คำตอบเดียวกัน อ่านน้อยกว่ามาก

## Mechanics

1. ทำให้แน่ใจว่าฟังก์ชันที่คุณกำลังจะแทนที่ถูกครอบคลุมด้วยเทสต์อย่างเต็มที่ รวมถึง edge case ของมัน — input ว่าง, ไม่พบ match, ค่าที่ขอบเขต ถ้าความครอบคลุมยังบาง ให้เพิ่มเทสต์ *ก่อน* และยืนยันว่าผ่านเมื่อทดสอบกับโค้ดที่มีอยู่
2. คิดอัลกอริทึมตัวแทนให้ครบถ้วนแล้วเขียนมัน ในอุดมคติคือเขียนเป็นฟังก์ชันแยกอยู่ข้างอันเดิม
3. เปรียบเทียบพฤติกรรมทั้งสอง สำหรับฟังก์ชันที่มี input space เล็กและนับได้ ให้รันทั้งสองอันกับทุก input แล้ว assert ว่าให้ผลตรงกัน มิฉะนั้นพึ่งพาชุดเทสต์ของคุณบวกกับ case เฉพาะเจาะจงไม่กี่กรณี
4. สลับ body เก่าเป็นอันใหม่
5. **รันเทสต์ของคุณ** ทุก case ต้องยังคงผ่าน
6. กำจัดอัลกอริทึมเก่าและ helper ใด ๆ ที่มันต้องการเมื่อไม่มีอะไรอ้างถึงพวกมันแล้ว

## เมื่อใดควรใช้ / ข้อแลกเปลี่ยน

Substitute Algorithm คุ้มค่าเมื่อมีวิธีที่ชัดเจนกว่าหรือเป็นมาตรฐานที่ให้ผลลัพธ์เดียวกัน — แทนที่ลูปที่ทำเองด้วยการเรียก library หรือเงื่อนไขที่แผ่ขยายด้วยการ lookup จากตาราง เวอร์ชันใหม่อ่านง่ายกว่าและมักจะขยายต่อได้ง่ายกว่า

ความเสี่ยงสูงกว่าการเคลื่อนไหวที่เล็กกว่าเพราะคุณกำลังเปลี่ยนกลยุทธ์ทั้งหมด ดังนั้นพฤติกรรมที่แยบยลอาจแตกต่างกัน — ลำดับการตัดสินเสมอที่ต่างกัน การจัดการค่าซ้ำหรือ input ว่างที่เปลี่ยนไป จงลองทำเฉพาะเมื่อมีเทสต์ที่หนักแน่น และเลือกที่จะแยกฟังก์ชันยักษ์ที่ซับซ้อนยุ่งเหยิงออกเป็นชิ้นเล็ก ๆ *ก่อน* ที่จะแทนที่ เพื่อให้การแทนที่แต่ละครั้งเล็กและตรวจสอบได้
