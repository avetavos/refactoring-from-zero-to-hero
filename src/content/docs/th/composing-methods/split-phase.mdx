## เจตนา

เมื่อโค้ดบล็อกหนึ่งทำสิ่งที่ต่างกันจริง ๆ สองอย่างตามลำดับ — เช่น *parse* ข้อมูลนำเข้าแล้ว *คำนวณ* ผลลัพธ์จากมัน — ให้แยกมันออกเป็นสอง phase phase แรกสร้างโครงสร้างข้อมูลกลางที่ชัดเจน phase ที่สองบริโภคเฉพาะโครงสร้างนั้น

## smell

นี่คือวิธีรักษา **tangled responsibilities** ภายในโค้ดช่วงเดียว การ parse และการคำนวณถูกสอดประสานกัน ดังนั้นการเปลี่ยน format ของ input ก็บังคับให้คุณต้องไปแตะคณิตศาสตร์ และการเปลี่ยนคณิตศาสตร์ก็บังคับให้คุณต้องคิดถึง string ทั้งสองความกังวลใช้ตัวแปรร่วมกัน และผู้อ่านไม่สามารถโฟกัสทีละอย่างได้เลย การตั้งชื่อโครงสร้างกลางขีดเส้นที่สะอาดระหว่างทั้งสอง

## Before → After

ฟังก์ชันหนึ่ง parse string `"name:qty:price"` ดิบและคำนวณยอดรวมในลมหายใจเดียว แยกมัน: parse ให้เป็น order object แล้วค่อยคิดราคา order

## Mechanics

1. extract phase ที่สอง (การคำนวณ) ออกมาเป็นฟังก์ชันของตัวเอง **รันเทสต์ของคุณ**
2. introduce โครงสร้างข้อมูลกลางแล้วส่งมันเข้าไปเป็น argument ให้ฟังก์ชัน phase ที่สองนั้น
3. ตรวจสอบแต่ละค่าที่ phase ที่สองอ่านจาก phase แรก ย้ายแต่ละค่าไปไว้บนโครงสร้างกลางทีละค่า โดยรันเทสต์หลังการย้ายแต่ละครั้ง
4. extract phase แรก (การ parse) ออกมาเป็นฟังก์ชันของตัวเองที่คืนโครงสร้างกลาง
5. ยืนยันว่าฟังก์ชันระดับบนสุดตอนนี้เป็นเพียง phase แรกป้อนเข้า phase ที่สอง แล้ว **รันชุดเทสต์ทั้งหมด**

## เมื่อใดควรใช้ / ข้อแลกเปลี่ยน

Split Phase เมื่อบล็อกหนึ่งจัดการสองขั้นตอนอย่างชัดเจน — parse แล้ว compute, validate แล้ว act, fetch แล้ว format — และขั้นตอนเหล่านั้นมีอะไรร่วมกันน้อยนอกเหนือจากการส่งต่อ โครงสร้างกลางกลายเป็น contract ที่มีชื่อระหว่างพวกมัน ดังนั้นแต่ละ phase เปลี่ยนแปลงได้อย่างอิสระและถูกเทสต์ได้ด้วยตัวมันเอง

ราคาที่ต้องจ่ายคือ data type พิเศษหนึ่งอันและ indirection เล็ก ๆ ถ้าสอง "phase" พันกันแน่นหรือสั้นแบบไม่สำคัญ รอยต่อก็เป็นของปลอม — ปล่อยให้โค้ดเป็นชิ้นเดียวไว้
