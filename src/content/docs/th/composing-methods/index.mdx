## Intent

กลุ่ม **composing methods** เป็นเรื่องของ *ภายใน* ของ function: statement ต่าง ๆ ถูกจัดกลุ่ม ตั้งชื่อ และเรียงลำดับอย่างไร refactoring เหล่านี้ไม่เปลี่ยน behavior แต่เปลี่ยนว่ามนุษย์จะติดตามสิ่งที่ function พยายามจะสื่อได้ง่ายเพียงใด

## Code Smell

function เติบโตขึ้นเมื่อเวลาผ่านไป มันสะสมtemporary variable ตรงนี้ เงื่อนไขบรรทัดเดียวอันแสนฉลาดตรงนั้น ลูปที่แอบทำงานสามอย่างเงียบ ๆ ในที่สุดก็ไม่มีใครอ่านมันจากบนลงล่างแล้วอธิบายได้ในประโยคเดียว วิธีรักษาในโมดูลนี้โจมตีปัญหานั้นโดยตรง: มันแยกชิ้นส่วนที่พันกันยุ่งเหยิงออกจากกัน ตั้งชื่อให้ส่วนสำคัญ และปล่อยให้ code ที่เหลืออ่านได้เหมือนสรุปย่อ ๆ ของเจตนา

| Code Smell | อาการที่บ่งชี้ | ท่าที่แนะนำ |
|---|---|---|
| Long Method | function ยาวจนอ่านจากบนลงล่างแล้วอธิบายไม่ได้ในประโยคเดียว | Extract Function |
| Duplicated Code | logic เดิมซ้ำในหลายจุด คัดลอก-วางมาพร้อมบั๊ก | Extract Function |
| Cryptic Expression | sub-expression ที่อ่านแล้วไม่รู้ความหมาย | Extract Variable |
| Temporary Variable | temporary variable ที่คำนวณค่าซับซ้อนแล้วส่งต่อ | Replace Temp with Query |
| Divergent Change | function เดียวทำสองงานที่ต่างกัน แก้ที่หนึ่งกระทบอีกที่ | Split Phase |

## โมดูลนี้ครอบคลุมอะไรบ้าง

refactoring เก้าแบบ แต่ละแบบเล็กและย้อนกลับได้:

- **[Extract Function](/refactoring/th/composing-methods/extract-function/)** — ยกชิ้นส่วนหนึ่งขึ้นไปเป็น function ของตัวเองที่มีชื่อดี เพื่อให้ผู้เรียกอ่านได้เหมือนรายการของเจตนา เป็นการเคลื่อนไหวพื้นฐานที่สุดในบรรดาทั้งหมด
- **[Inline Function](/refactoring/th/composing-methods/inline-function/)** — สิ่งที่ตรงกันข้าม: เมื่อ body ของ helper เรียบง่ายพอ ๆ กับชื่อของมัน ก็พับมันกลับเข้าไป
- **[Extract Variable](/refactoring/th/composing-methods/extract-variable/)** — ตั้งชื่อให้ sub-expression ที่ชวนสับสน เพื่อให้บรรทัดนั้นอธิบายตัวเองได้
- **[Inline Variable](/refactoring/th/composing-methods/inline-variable/)** — สิ่งที่ตรงกันข้าม: ลบตัวแปรที่เป็นเพียงเสียงสะท้อนของ expression ของมันและไม่ได้เพิ่มความชัดเจนใด ๆ
- **[Replace Temp with Query](/refactoring/th/composing-methods/replace-temp-with-query/)** — เปลี่ยนตัวแปร local ที่เก็บค่าที่คำนวณไว้ให้กลายเป็น function เพื่อให้การคำนวณนำกลับมาใช้ซ้ำได้และ extract ได้ง่าย
- **[Slide Statements](/refactoring/th/composing-methods/slide-statements/)** — ย้าย statement ที่เกี่ยวข้องกันมาอยู่ด้วยกัน เพื่อให้ชิ้นส่วนที่สอดคล้องกันตั้งพร้อมให้อ่านหรือ extract
- **[Split Phase](/refactoring/th/composing-methods/split-phase/)** — เมื่อบล็อกหนึ่งทำงานสองอย่างที่ต่างกัน ก็แยกมันออกเป็น phase ที่เรียงลำดับและเชื่อมกันด้วยค่ากลางที่ชัดเจน
- **[Substitute Algorithm](/refactoring/th/composing-methods/substitute-algorithm/)** — แทนที่วิธีทำบางอย่างที่ซับซ้อนยุ่งเหยิงด้วยวิธีที่ชัดเจนกว่าซึ่งให้ผลลัพธ์เดียวกัน
- **[Replace Loop with Pipeline](/refactoring/th/composing-methods/replace-loop-with-pipeline/)** — แปลงลูปแบบ imperative ให้เป็นโซ่ของขั้นตอนที่มีชื่อ — filter, map, reduce

แต่ละบทเรียนแสดงตัวอย่าง before-and-after เดียวกันในภาษา TypeScript, Python, Go และ Rust จากนั้นพาเดินผ่าน mechanics ที่ปลอดภัยและเรียงลำดับ — โดยรันเทสต์ของคุณระหว่างทุกขั้นตอน

<Callout title="ออกแบบมาให้รักษา behavior">

</Callout>
