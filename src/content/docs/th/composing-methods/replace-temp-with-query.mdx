## เจตนา

นำตัวแปร local ที่ถูกกำหนดค่าด้วยผลของการคำนวณแล้วถูกอ่านอย่างเดียว มาแทนที่ด้วยฟังก์ชัน (เรียกว่า "query") ที่ทำการคำนวณนั้น ทุกที่ที่เคยใช้ temp ตอนนี้เรียก query แทน

## smell

นี่คือวิธีรักษา **temp ที่ตรึง logic ไว้กับที่** ตัวแปรชั่วคราวผูกการคำนวณไว้กับฟังก์ชันเดียว: คุณนำผลลัพธ์ไปใช้ซ้ำที่อื่นไม่ได้ และคุณ Extract Function รอบโค้ดที่พึ่งพามันไม่ได้โดยไม่ลาก temp ไปด้วยในฐานะ parameter การเปลี่ยน temp ให้เป็น query ปลดปล่อยการคำนวณนั้น มันมีค่าที่สุดในฐานะขั้นตอน *เตรียมความพร้อม* — เมื่อค่ากลายเป็น method แล้ว การ extract ที่ใหญ่กว่าก็หลุดออกมาอย่างสะอาด

## Mechanics

1. ตรวจสอบว่า temp ถูกกำหนดค่าเพียงครั้งเดียวและไม่เคยถูก mutate หลังการกำหนดค่า ถ้ามันถูกกำหนดค่าใหม่ ให้แยกมันก่อน เพื่อให้แต่ละค่าถูกกำหนดในที่เดียว
2. ยืนยันว่าการคำนวณฝั่งขวามือไม่มี side effect
3. extract การคำนวณออกมาเป็นฟังก์ชัน query (เป็น method เมื่อข้อมูลอยู่บน object)
4. แทนที่การอ่าน temp แต่ละครั้งด้วยการเรียก query
5. **รันเทสต์ของคุณ** หลังการแทนที่แต่ละครั้ง
6. ลบการประกาศ temp ที่ตอนนี้ไม่ได้ใช้แล้วทิ้ง

## เมื่อใดควรใช้ / ข้อแลกเปลี่ยน

ใช้มันเมื่อ temp ขวาง Extract Function ที่คุณอยากทำ เมื่อการคำนวณเดียวกันจะมีประโยชน์จากมากกว่าหนึ่ง method หรือเพียงเพื่อลด local state ของ method ที่ยาว ข้อแลกเปลี่ยนหลักคือการคำนวณซ้ำ: query อาจคำนวณใหม่ในทุกการเรียก สำหรับ business logic ทั่วไปสิ่งนี้แทบไม่มีผล และความชัดเจนก็คุ้มค่า สำหรับ hot path ที่ร้อนจริง ๆ ให้วัดก่อนตัดสินใจ

ทิศทางตรงข้ามคือ cache ค่ากลับเข้าไปใน local เมื่อการ profiling พิสูจน์ว่า query เป็นคอขวดจริง — แต่เอื้อมไปหาสิ่งนั้นเมื่อมีหลักฐานเท่านั้น
