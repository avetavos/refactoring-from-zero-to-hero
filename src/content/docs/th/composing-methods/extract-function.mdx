## เจตนา

นำชิ้นส่วนโค้ดที่เป็นเอกภาพชิ้นหนึ่งมาย้ายเข้าไปในฟังก์ชันใหม่ แล้วตั้งชื่อฟังก์ชันนั้นตาม *จุดประสงค์* ที่มันทำหน้าที่อยู่ จุดเดิมตอนนี้เพียงแค่เรียก helper จึงอ่านได้เหมือนรายการสั้น ๆ ของขั้นตอนที่มีชื่อ แทนที่จะเป็นกำแพงของรายละเอียด

## smell

นี่คือวิธีรักษา **Long Function** — และรักษาบล็อกโค้ดใด ๆ ที่ต้องมี comment มาอธิบายว่ามันทำอะไร comment ที่เขียนว่า "ตอนนี้คำนวณภาษี" ไว้เหนือโค้ดสิบบรรทัด คือป้ายไฟกระพริบที่บอกว่าโค้ดสิบบรรทัดนั้นอยากเป็นฟังก์ชันชื่อ `taxFor` เมื่อชื่อฟังก์ชันแบกเจตนาไว้ comment ก็กลายเป็นสิ่งไม่จำเป็น และผู้เรียกก็อ่านง่าย

## Before → After

ฟังก์ชันใบเสร็จที่คำนวณ subtotal นำภาษีมาคิด แล้วพิมพ์ออกมา ก่อนหน้านี้มันทำทุกอย่างแบบ inline หลังจากนั้น แต่ละงานกลายเป็น helper ที่มีชื่อ

## Mechanics

1. ตัดสินใจเลือกชิ้นส่วนที่จะ extract และชื่อที่บอกว่า *มันทำสำเร็จอะไร* ไม่ใช่บอกว่าทำอย่างไร ถ้าคุณตั้งชื่อให้มันสะอาด ๆ ไม่ได้ ชิ้นส่วนนั้นอาจยังไม่เป็นหน่วยที่เป็นเอกภาพ — ลองปรับขอบเขตของมันใหม่
2. สร้างฟังก์ชันว่างใหม่ที่มีชื่อนั้น
3. คัดลอกชิ้นส่วนนั้นเข้าไปในฟังก์ชันใหม่
4. ดูตัวแปรที่ชิ้นส่วนนั้นอ่าน: ส่งมันเข้าไปเป็น parameter ดูตัวแปรที่มันสร้างขึ้นและโค้ดภายนอกยังต้องใช้: คืนค่ามันออกมา
5. แทนที่ชิ้นส่วนเดิมด้วยการเรียกฟังก์ชันใหม่
6. **รันเทสต์ของคุณ** พฤติกรรมต้องไม่เปลี่ยนแปลง
7. ทำซ้ำกับชิ้นส่วนถัดไป ให้แต่ละการ extract เล็ก ๆ เพื่อให้เทสต์ที่ล้มเหลวชี้ไปยังการเปลี่ยนแปลงเล็ก ๆ เพียงจุดเดียว

## เมื่อใดควรใช้ / ข้อแลกเปลี่ยน

เอื้อมไปหา Extract Function เมื่อบล็อกหนึ่งมีจุดประสงค์เดียวที่ชัดเจน เมื่อคุณรู้สึกอยากเขียน comment มาอธิบาย หรือเมื่อคุณเห็นชิ้นส่วนเดียวกันถูกทำซ้ำในสองที่ — extract เพียงครั้งเดียวแล้วทั้งสองจุดก็เรียก helper ที่ใช้ร่วมกันได้

ราคาที่ต้องจ่ายคือการกระโดดเล็ก ๆ ในการนำทาง: ผู้อ่านต้องตามชื่อไปยังนิยามของมัน แต่การแลกเปลี่ยนนั้นแทบจะคุ้มเสมอ เพราะชื่อที่ดีทำให้ผู้อ่านส่วนใหญ่หยุดได้ *โดยไม่ต้อง* ตามการกระโดดนั้นไป refactoring ที่ตรงกันข้ามคือ **Inline Function**: ถ้าชื่อของ helper บอกอะไรไม่มากไปกว่า body ของมัน ก็พับมันกลับเข้าไป
