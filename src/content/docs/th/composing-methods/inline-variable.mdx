## เจตนา

บางครั้งตัวแปร local ก็เป็นเพียงชื่อที่สองของ expression ที่คุณเข้าใจอยู่แล้ว มันไม่ได้ทำให้อะไรชัดเจนขึ้นและไม่ได้ถูกนำกลับมาใช้ซ้ำ — มันแค่ขวางอยู่ระหว่างคุณกับโค้ดที่สำคัญ Inline Variable กำจัดตัวกลางนั้น: แทนที่ตัวแปรด้วย expression แล้วลบการประกาศตัวแปรทิ้ง ผลลัพธ์คือชื่อที่ต้องคอยติดตามน้อยลงหนึ่งชื่อ

## smell

คุณเจอตัวแปรที่ชื่อของมันไม่ได้บอกอะไรมากไปกว่า expression ที่มันเก็บไว้ `let basePrice = order.basePrice;` ตามด้วยการใช้ `basePrice` เพียงครั้งเดียวคือเสียงรบกวน — ผู้อ่านต้องเลื่อนขึ้นไปยืนยันว่า `basePrice` คือ `order.basePrice` จริง ๆ และไม่มีอะไรมากกว่านั้น ตัวแปรนั้นสัญญาว่าจะอธิบายแต่กลับให้เพียงคำพ้องความหมาย

นี่คือสิ่งที่ตรงกันข้ามกับ **Extract Variable** อย่างชัดเจน ที่นั่นคุณเพิ่มชื่อเพราะ sub-expression ลึกลับ ที่นี่คุณกำจัดชื่อเพราะมันไม่เคยคุ้มค่ากับที่ของมัน การตัดสินใจเหมือนกันทั้งสองทิศทาง: ชื่อนั้นทำให้โค้ดชัดเจนกว่า expression ดิบหรือไม่? ถ้าไม่ ก็ inline มันเสีย

## Before → After

การตรวจสอบเรื่องการจัดส่งที่ผูกตัวแปรสองตัวซึ่งเป็นเพียงภาพสะท้อนของฟิลด์ แล้วใช้แต่ละตัวเพียงครั้งเดียว

## Mechanics

1. ตรวจสอบว่า expression ที่กำหนดให้กับตัวแปรไม่มี side effect — การ inline ต้องไม่เปลี่ยน *เมื่อใด* หรือ *บ่อยแค่ไหน* ที่ expression นั้นทำงาน
2. ถ้าตัวแปรยังไม่ใช่ read-only ให้ทำให้มันเป็นอย่างนั้น (เป็น `const`, `final`, หรือ binding ที่กำหนดค่าได้ครั้งเดียว) ถ้า compiler บ่น แสดงว่าตัวแปรถูกกำหนดค่าใหม่ และคุณควรหยุด: มันทำมากกว่าการสะท้อนภาพ
3. หาที่แรกที่ตัวแปรถูกอ่าน แล้วแทนที่การอ่านนั้นด้วย expression
4. **รันเทสต์ของคุณ**
5. ทำซ้ำสำหรับการอ่านที่เหลือแต่ละครั้ง ทีละครั้ง โดยทดสอบไปตลอดทาง
6. เมื่อไม่มีการอ่านเหลืออยู่ ให้ลบการประกาศตัวแปรทิ้ง
7. **รันเทสต์ของคุณ** อีกครั้ง

## เมื่อใดควรใช้ / ข้อแลกเปลี่ยน

inline ตัวแปรเมื่อชื่อของมันไม่ได้พกพาข้อมูลที่ expression ไม่ได้พกพาอยู่แล้ว และเมื่อมันถูกใช้ในเพียงหนึ่งหรือสองที่ที่อยู่ใกล้กัน การถอดมันออกทำให้ฟังก์ชันสั้นลงและกำจัดการกระโดดไปมาหนึ่งครั้งสำหรับผู้อ่าน

อย่า inline เมื่อชื่อของตัวแปรอธิบายความหมายที่ไม่ชัดเจนในตัว เมื่อ expression ถูกใช้ซ้ำหลายครั้ง (ชื่อจะช่วยหลีกเลี่ยงทั้งความซ้ำซ้อนและการคำนวณซ้ำ) หรือเมื่อ expression มีต้นทุนสูงและตัวแปรเก็บผลลัพธ์ไว้เป็น cache ในกรณีเหล่านั้น สิ่งที่ตรงกันข้าม — **Extract Variable** — คือการเคลื่อนไหวที่คุณต้องการแทน
