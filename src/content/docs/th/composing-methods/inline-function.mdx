## Intent

แทนที่การเรียก function เล็ก ๆ ด้วย body ของ function นั้น แล้วลบ function ทิ้ง ใช้มันเมื่อชื่อไม่ได้เพิ่มอะไรที่ body ยังไม่ได้ทำให้ชัดเจนอยู่แล้ว

## Code Smell

นี่คือวิธีรักษา **needless indirection** บางครั้ง helper เคยสมเหตุสมผลในอดีต แต่หดเหลือเพียงบรรทัดเดียวที่ไม่สำคัญ หรือชื่อของมันก็เพียงแค่พูดซ้ำสิ่งที่ statement เดียวของมันทำ การตามชื่อไปยังนิยามที่บอกอะไรไม่มากไปกว่าจุดที่เรียก คือความเหนื่อยเปล่าสำหรับผู้อ่าน การ inline กำจัดการกระโดดนั้น มันยังเป็นขั้นตอนทำความสะอาดที่มีประโยชน์: ยุบ helper เล็ก ๆ ที่พันกันยุ่งเหยิงให้มาอยู่ที่เดียว แล้วค่อย extract ขอบเขตที่ดีกว่าออกมาใหม่

## Before → After

helper ชื่อ `ratingFor` ที่เพียงแค่ส่งต่อไปยังการตรวจสอบบรรทัดเดียว เพิ่มชั้นหนึ่งโดยไม่ได้ผลตอบแทนอะไร inline มันเสีย

## Mechanics

1. ยืนยันว่า function ไม่เป็น polymorphic — คุณ inline method ที่ subclass override ไม่ได้
2. ค้นหาทุกจุดที่เรียก (ถ้ามีมาก ให้ทบทวนใหม่ การ inline ง่ายที่สุดเมื่อจุดเรียกมีไม่กี่จุด)
3. แทนที่การเรียกแต่ละครั้งด้วยสำเนาของ body function โดยปรับให้เข้ากับ argument จริง
4. **รันเทสต์ของคุณ** หลังการแทนที่แต่ละครั้ง
5. เมื่อไม่เหลือจุดเรียกแล้ว ให้ลบ function เดิมทิ้ง
6. รันชุดเทสต์ทั้งหมดอีกครั้งเพื่อยืนยันว่าไม่มีอะไรอ้างอิงถึงมัน

## When to use / trade-offs

inline เมื่อ body ชัดเจนอย่างน้อยเท่ากับชื่อ เมื่อกลุ่มของ helper ที่จัดโครงสร้างไม่ดีควรถูกรวมก่อนจะแยกใหม่ หรือเมื่อ indirection ขวางทางผู้อ่าน **อย่า** inline function ที่ถูกเรียกจากหลายที่ หรือ function ที่ชื่อจับใจความของแนวคิดที่ไม่ชัดเจนได้จริง ๆ — ในกรณีนั้นชื่อคือเอกสารที่ควรเก็บไว้

สิ่งที่ตรงกันข้ามคือ **Extract Function** ทั้งสองเป็นคู่หูกัน: inline เพื่อยุบโครงสร้างที่ไม่ดี แล้ว extract เพื่อสร้างโครงสร้างที่ดีกว่าขึ้นมาใหม่

| ใช้ Inline Function เมื่อ | หลีกเลี่ยงเมื่อ |
|---|---|
| ชื่อ function ไม่ได้บอกอะไรมากกว่า body | function นั้นถูกใช้งานในหลายที่ |
| function มีแค่ delegation ไปยังอีก function | body ซับซ้อนกว่าชื่อ — ชื่อยังมีประโยชน์ |
| กำลัง re-organize และต้องการรวม logic ก่อนแบ่งใหม่ | ต้องการ override ใน subclass |

> ⚠️ ไม่ควร Inline Function เมื่อ:
> - function นั้นซ่อน complexity จริง ๆ — inlining จะทำ code ยากขึ้น
> - เป็น API ที่ external caller ใช้งาน
> - ต้องการ trace เพื่อ debug — ชื่อ function ช่วยใน stack trace

<Callout title="Inline คือปุ่มรีเซ็ต">
เมื่อขอบเขตของ helper ผิด อย่าแก้ไขมันตรงนั้น inline ทุกอย่างกลับเข้าไปใน function เดียว มองเห็น flow ทั้งหมดอย่างชัดเจน แล้วค่อย Extract ตามขอบเขตที่สำคัญจริง ๆ การ inline มักเป็น *วิธีการ* ไปสู่การ extract ที่สะอาดกว่า
</Callout>
