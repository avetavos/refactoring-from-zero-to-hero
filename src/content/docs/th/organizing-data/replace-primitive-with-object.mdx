## เจตนา

นำค่า primitive — สตริง ตัวเลข — ที่เริ่มสะสมกฎเกี่ยวกับวิธีตรวจสอบความถูกต้อง วิธี parse หรือวิธีจัดรูปแบบ แล้วห่อมันไว้ใน type ขนาดเล็กที่อุทิศไว้โดยเฉพาะ value object ใหม่ถือข้อมูลดิบ *และ* พฤติกรรม ดังนั้นกฎจึงอยู่ในจุดเดียว และชื่อ type ก็ประกาศว่าค่านั้นแท้จริงแล้วคืออะไร

## กลิ่นเหม็น

นี่คือยารักษาอาการ **Primitive Obsession**: การแทนแนวคิดเชิงโดเมนที่มีความหมายด้วย primitive ดิบ ๆ หมายเลขโทรศัพท์เป็น `string` จำนวนเงินเป็น `number` อุณหภูมิเป็น `float` เปลือย ๆ อาการเหมือนกัน คือลอจิกเดียวกันกระจัดกระจายไปทุกที่ที่มีการใช้ primitive — regex เดิมในการตรวจสอบหมายเลขโทรศัพท์ การปัดเศษเดิมในการจัดการเซนต์ การแปลงเดิมที่ทำซ้ำที่แต่ละจุด การทำซ้ำแต่ละครั้งคือจุดที่จะลืมกฎและสร้างบั๊กขึ้นมา

## ก่อน → หลัง

หมายเลขโทรศัพท์ที่ถูกพกพาเป็นสตริงดิบ พร้อมการตรวจสอบความถูกต้องและการจัดรูปแบบที่ทำซ้ำในทุกการใช้งาน หลังจากนั้น value type `PhoneNumber` ขนาดเล็กเป็นเจ้าของตัวเลขและพฤติกรรม

    // Before
    fn call_customer(phone: &str) -> Result<(), String> {
        let digits: String = phone.chars().filter(|c| c.is_ascii_digit()).collect();
        if digits.len() != 10 {

## กลไก

1. สร้างคลาสใหม่ที่มีฟิลด์เดียวถือ primitive ไว้ ให้คอนสตรักเตอร์ (หรือฟังก์ชัน parse) ที่รับค่าดิบและตรวจสอบความถูกต้องเพียงครั้งเดียว
2. ย้ายพฤติกรรมที่กระจัดกระจายชิ้นหนึ่ง — การตรวจสอบความถูกต้อง การจัดรูปแบบ การ parse — เข้าไปเป็นเมท็อดบน type ใหม่
3. แทนที่การใช้งาน primitive ดิบหนึ่งจุดด้วย value object โดยเรียกเมท็อดใหม่แทนลอจิกที่เขียนอินไลน์
4. **รันเทสต์ของคุณ**
5. ทำซ้ำ: ย้ายแต่ละจุดใช้งานและรวมแต่ละกฎที่ซ้ำกันเข้าเป็นเมท็อด ทีละอัน
6. เมื่อทุกจุดใช้ value object แล้ว primitive และลอจิกที่กระจัดกระจายของมันจะหายไปจากโค้ดที่เรียกใช้ ตอนนี้ type เป็นบ้านของกฎใด ๆ ในอนาคตเกี่ยวกับแนวคิดนั้น

## เมื่อไรควรใช้ / ข้อแลกเปลี่ยน

ยกระดับ primitive เมื่อมันแบกความหมายเกินกว่า type ดิบของมัน และเมื่อคุณพบว่าตัวเองทำลอจิกเกี่ยวกับมันซ้ำ ๆ — การตรวจสอบความถูกต้อง การเปรียบเทียบ การจัดรูปแบบ การแปลงหน่วย value object รวมศูนย์กฎเหล่านั้น ทำให้สถานะที่ไม่ถูกต้องสร้างได้ยากขึ้น และให้ระบบ type ดักจับ `PhoneNumber` ที่ถูกส่งเข้าไปในที่ที่คาดหวัง `ZipCode` ได้

ต้นทุนคือคลาสเพิ่มอีกหนึ่งและงานในการสร้างมันที่ขอบเขต สำหรับค่าที่ใช้ในที่เดียวโดยไม่มีกฎผูกติด primitive ดิบก็เพียงพอดีอยู่แล้ว — การห่อมันจะเป็นพิธีรีตองที่ไม่ได้ผลตอบแทน สัญญาณที่ให้ลงมือทำคือ *การทำซ้ำ*: สำเนาที่สองหรือที่สามของกฎเดียวกันคือช่วงเวลาที่ object เริ่มคุ้มค่า
