## เจตนา

ตระกูล **organizing data** ว่าด้วยเรื่องของ *รูปร่าง* ของค่าต่าง ๆ ที่โปรแกรมพกพาไปมา และ *เส้นทาง* ที่โค้ดใช้เข้าถึงค่าเหล่านั้น โปรแกรมจะชัดเจนได้แค่เพียงเท่าที่ข้อมูลของมันชัดเจน เมื่อฟิลด์หนึ่งสามารถถูกแตะต้องได้จากทุกที่ เมื่อค่า `0.07` ดิบ ๆ ล่องลอยผ่านสามโมดูลโดยไม่มีชื่อ หรือเมื่อสตริงธรรมดา ๆ แอบแบกกฎเกี่ยวกับวิธีจัดรูปแบบของตัวมันเองไว้ ข้อมูลนั้นกำลังทำงานต่อต้านคุณ การรีแฟคเตอร์เหล่านี้สร้างขอบเขตล้อมรอบข้อมูล ให้ชื่อกับมัน และปล่อยให้มันเติบโตขึ้นพร้อมพฤติกรรมที่มันเงียบ ๆ ร้องขอมาตลอด

## กลิ่นเหม็น

ข้อมูลที่ยุ่งเหยิงปรากฏให้เห็นในรูปแบบเล็ก ๆ ที่เกิดซ้ำ ๆ ค่าคงที่ตัวเดียวกันโผล่ขึ้นในห้าไฟล์และไม่มีใครจำได้ว่ามันหมายถึงอะไร ฟิลด์ที่เป็นลิสต์ถูกส่งต่อออกไป แล้วผู้เรียกที่อยู่ไกลออกไปก็ `push` ลงไป ทำให้ค่าคงตัว (invariant) ที่เจ้าของไม่เคยตกลงด้วยพังลง หมายเลขโทรศัพท์ถูกเก็บเป็น `string` ดังนั้นทุกการตรวจสอบความถูกต้องและทุกการจัดรูปแบบใหม่จึงถูกคัดลอกวางซ้ำ ๆ ที่จุดใช้งานแต่ละแห่ง สิ่งเหล่านี้ไม่มีอันไหนรุนแรงเลย แต่เมื่อรวมกันแล้วมันทำให้โค้ดเบสเปราะบาง การเปลี่ยน *วิธี* ที่ข้อมูลถูกจัดเก็บจะส่งผลกระทบกระเพื่อมไปทุกที่ที่อ่านมันโดยตรง

## โมดูลนี้ครอบคลุมอะไรบ้าง

สิบการรีแฟคเตอร์ แต่ละอันกระชับความสัมพันธ์ระหว่างข้อมูลกับโค้ดที่ใช้มัน:

- **[Encapsulate Variable](/refactoring-from-zero-to-hero/th/organizing-data/encapsulate-variable/)** — ห่อหุ้มการเข้าถึงข้อมูลโดยตรงไว้หลังฟังก์ชัน accessor เพื่อให้คุณมีจุดเดียวสำหรับตรวจสอบความถูกต้อง บันทึก log หรือเปลี่ยนวิธีแทนค่าในภายหลัง
- **[Replace Magic Literal](/refactoring-from-zero-to-hero/th/organizing-data/replace-magic-literal/)** — ให้ชื่อเชิงสัญลักษณ์แก่ตัวเลขหรือสตริงที่ไม่มีคำอธิบาย เพื่อให้ค่าคงที่อธิบายความหมายของตัวเอง และเปลี่ยนแปลงได้ในจุดเดียว
- **[Encapsulate Collection](/refactoring-from-zero-to-hero/th/organizing-data/encapsulate-collection/)** — หยุดเปิดเผยฟิลด์ลิสต์หรือแมปที่เปลี่ยนแปลงได้ คืนค่าเป็น view แบบอ่านอย่างเดียว และเสนอเมท็อด add กับ remove อย่างชัดเจน เพื่อให้เจ้าของยังควบคุมเนื้อหาของมันได้
- **[Replace Primitive with Object](/refactoring-from-zero-to-hero/th/organizing-data/replace-primitive-with-object/)** — เมื่อค่า primitive เริ่มดึงดูดกฎการตรวจสอบความถูกต้องและการจัดรูปแบบเข้ามา ให้ยกระดับมันขึ้นเป็น value type ขนาดเล็กที่ถือทั้งข้อมูลและพฤติกรรมของมัน
- **[Replace Type Code with Subclasses](/refactoring-from-zero-to-hero/th/organizing-data/replace-type-code-with-subclasses/)** — เมื่อฟิลด์ Type Code ขับเคลื่อนการแตกกิ่งแบบ `switch` ให้แทนที่ลอจิกเงื่อนไขด้วย polymorphism ที่กระจายอยู่ทั่วลำดับชั้นคลาสขนาดเล็ก
- **[Self-Encapsulate Field](/refactoring-from-zero-to-hero/th/organizing-data/self-encapsulate-field/)** — เดินทางแม้แต่การอ่านและเขียนฟิลด์ของคลาสตัวมันเองผ่าน getter และ setter ของฟิลด์นั้น เพื่อให้ subclass และการตรวจสอบความถูกต้องมีจุดเดียวให้เกาะเกี่ยว
- **[Split Variable](/refactoring-from-zero-to-hero/th/organizing-data/split-variable/)** — เมื่อตัวแปรหนึ่งถูกกำหนดค่าใหม่เพื่อหมายถึงสองสิ่งที่ต่างกัน ให้แต่ละจุดประสงค์มีตัวแปรที่กำหนดค่าครั้งเดียวเป็นของตัวเอง
- **[Rename Field](/refactoring-from-zero-to-hero/th/organizing-data/rename-field/)** — เมื่อชื่อฟิลด์ของ record หรือ struct ไม่เข้ากับข้อมูลของมันอีกต่อไป ให้เปลี่ยนชื่อมันและอัปเดตผู้ใช้ทุกราย เพื่อให้ชื่อกลับมาบอกความจริงอีกครั้ง
- **[Change Reference to Value (and back)](/refactoring-from-zero-to-hero/th/organizing-data/change-reference-to-value/)** — ทำให้อ็อบเจกต์ขนาดเล็กเป็น value ที่ไม่เปลี่ยนแปลงและเปรียบเทียบกันด้วยเนื้อหา เมื่อการแบ่งปันไม่ได้ให้ประโยชน์อะไร หรือกลับท่าตรงข้ามเมื่อหลาย ๆ ที่ต้องแบ่งปันอินสแตนซ์เดียวที่อัปเดตได้
- **[Replace Type Code with State/Strategy](/refactoring-from-zero-to-hero/th/organizing-data/replace-type-code-with-state-strategy/)** — เมื่อ Type Code ขับเคลื่อนพฤติกรรมและเปลี่ยนแปลงได้ขณะรันไทม์ ให้ถืออ็อบเจกต์ State หรือ Strategy ที่สลับเปลี่ยนได้ แทนที่จะใช้ subclass ตายตัว

แต่ละบทเรียนแสดงรูปแบบก่อน-และ-หลังเดียวกันในภาษา TypeScript, Python, Go และ Rust จากนั้นจึงเดินผ่านกลไกที่ปลอดภัยและเป็นลำดับ — โดยรันเทสต์ของคุณระหว่างทุกขั้นตอน
