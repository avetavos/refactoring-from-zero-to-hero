## เจตนา

**Replace Subclass with Fields** สลายลำดับชั้นเล็ก ๆ ที่ subclass ไม่มีพฤติกรรมใด ๆ — มีเพียง *ค่าคงที่ที่ต่างกัน* ที่ถูกคืนกลับจากเมท็อดที่ override เท่านั้น แต่ละเมท็อดแบบนั้นกลายเป็นฟิลด์บนคลาส concrete เดียว และค่าที่แยกความต่างของ subclass ก็ย้ายเข้าไปอยู่ใน constructor ของคลาสนั้น ลำดับชั้นหายไป ส่วนข้อมูลที่มันเข้ารหัสไว้ยังคงอยู่ในรูปฟิลด์ธรรมดา

## กลิ่นไม่ดี

คุณเจอ subclass ที่ทุก override เป็นบรรทัดเดียวคืนค่า literal: `isMale() { return true; }`, `code() { return "M"; }` subclass นั้นไม่เพิ่มตรรกะใด ไม่ถือ state ต่อ instance และไม่เคยเปลี่ยนคำตอบ ทั้ง type — พร้อมไฟล์ของตัวเอง constructor ของตัวเอง ช่องของตัวเองในลำดับชั้น — มีอยู่เพียงเพื่อ hard-code ค่าคงที่สองค่า นั่นคือ **Lazy Class** ที่แต่งตัวเป็น subtype และมันทำให้ความแตกต่างระหว่าง variant มองเห็นได้ยากกว่าการเขียนฟิลด์เรียงกันเป็นแถว

## Before → After

`Person` ถูกแยกเป็น subclass `Male` กับ `Female` แต่ละตัว override เมท็อดเพื่อคืนค่าคงที่ตายตัว หลังจากนั้น `Person` เดียวถือค่าคงที่เหล่านั้นไว้เป็นฟิลด์ที่ factory ตั้งให้

<Tabs syncKey="lang">
  <TabItem label="TypeScript">
    ```ts
    // Before — subclasses that only return constants
    abstract class Person {
      abstract isMale(): boolean;
      abstract code(): string;
    }

    class Male extends Person {
      isMale(): boolean { return true; }
      code(): string { return "M"; }
    }

    class Female extends Person {
      isMale(): boolean { return false; }
      code(): string { return "F"; }
    }

    // After — one class, the constants become fields
    class Person {
      private constructor(
        private readonly male: boolean,
        private readonly genderCode: string,
      ) {}

      static createMale(): Person { return new Person(true, "M"); }
      static createFemale(): Person { return new Person(false, "F"); }

      isMale(): boolean { return this.male; }
      code(): string { return this.genderCode; }
    }
    ```
  </TabItem>
  <TabItem label="Python">
    ```python
    # Before — subclasses that only return constants
    class Person:
        def is_male(self):
            raise NotImplementedError

        def code(self):
            raise NotImplementedError

    class Male(Person):
        def is_male(self):
            return True

        def code(self):
            return "M"

    class Female(Person):
        def is_male(self):
            return False

        def code(self):
            return "F"

    # After — one class, the constants become fields
    class Person:
        def __init__(self, male, code):
            self._male = male
            self._code = code

        @classmethod
        def create_male(cls):
            return cls(True, "M")

        @classmethod
        def create_female(cls):
            return cls(False, "F")

        def is_male(self):
            return self._male

        def code(self):
            return self._code
    ```
  </TabItem>
  <TabItem label="Go">
    ```go
    // Go has no inheritance, so the "before" would already be modelled with an
    // interface plus two empty structs whose methods return constants — pure
    // boilerplate. The idiomatic shape is the "after": one struct with fields,
    // built by constructor functions.

    type Person struct {
        male bool
        code string
    }

    func NewMale() Person   { return Person{male: true, code: "M"} }
    func NewFemale() Person { return Person{male: false, code: "F"} }

    func (p Person) IsMale() bool { return p.male }
    func (p Person) Code() string { return p.code }
    ```

    ```rust
    // Rust has no inheritance. The "before" — a trait with two unit structs
    // whose impls return constants — adds nothing but ceremony. The idiomatic
    // form holds the data in fields on one struct, with constructor functions.

    struct Person {
        male: bool,
        code: char,
    }

    impl Person {
        fn new_male() -> Self {
            Person { male: true, code: 'M' }
        }

        fn new_female() -> Self {
            Person { male: false, code: 'F' }
        }

        fn is_male(&self) -> bool {
            self.male
        }

        fn code(&self) -> char {
            self.code
        }
    }
    ```

## กลไก

1. ใช้ **Replace Constructor with Factory Function** กับ superclass เพื่อให้ผู้เรียกสร้าง instance ผ่านเมท็อด factory แทนการเรียก `new Subclass()` ตรง ๆ
2. สำหรับแต่ละเมท็อดที่ subclass override เพื่อคืนค่าคงที่ ให้เพิ่มฟิลด์บน superclass ที่ถือค่านั้นไว้
3. ทำให้แต่ละเมท็อด factory ส่งค่าคงที่ที่เหมาะสมตอนสร้าง instance ของ superclass
4. เปลี่ยนเมท็อดของ superclass ให้คืนค่าฟิลด์ใหม่แทนการเป็น abstract
5. **รันเทสต์ของคุณ** ทุกจุดเรียกควรสร้าง instance ของ superclass ด้วยค่าฟิลด์ที่ถูกต้องแล้ว
6. ลบ subclass ที่ว่างเปล่าทิ้งเมื่อไม่มีอะไรอ้างถึงมันแล้ว

## เมื่อไรควรใช้ / ข้อแลกเปลี่ยน

หยิบสิ่งนี้มาใช้เมื่อใดก็ตามที่งานเดียวของ subclass คือการ hard-code ค่าคืนกลับ — ไม่มีฟิลด์เพิ่ม ไม่มีพฤติกรรมจริง ไม่มีตรรกะต่อการเรียก การพับค่าเหล่านั้นเข้าไปเป็นฟิลด์จะลบ type หนึ่งออก ทำให้ลำดับชั้นแบนลง และจัดเรียง variant ให้ความแตกต่างของมันอ่านได้เป็นข้อมูลแทนที่จะเป็นนิยามคลาส

*อย่า* นำมาใช้ถ้า subclass มีพฤติกรรมจริงที่แตกต่างกัน หรือถ้าคุณคาดว่ามันจะงอกพฤติกรรมขึ้นมา เพราะถ้าเป็นเช่นนั้น ลำดับชั้น (หรือ strategy) ก็ยังคุ้มค่าที่จะมีอยู่ สิ่งตรงข้ามคือ **Replace Fields with Subclass** — เลื่อนฟิลด์ที่แยกความต่างกลับขึ้นไปเป็น subtype เมื่อพฤติกรรม ไม่ใช่เพียงข้อมูล เริ่มแยกตัวออกตามฟิลด์นั้น
