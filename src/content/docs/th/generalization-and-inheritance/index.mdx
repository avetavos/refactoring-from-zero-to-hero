## เจตนา

กลุ่มของ **generalization and inheritance** ย้ายฟีเจอร์ *ในแนวตั้ง* และ *ในแนวข้าง* ข้ามไปมาในลำดับชั้นของ type ที่เกี่ยวข้องกัน สมาชิกที่พี่น้องสองคลาสมีร่วมกันถูกดึงขึ้นไปไว้ที่พ่อแม่ สมาชิกที่มีเพียงสาขาเดียวต้องใช้ถูกผลักลงไป สองคลาสที่ไม่เกี่ยวข้องกันแต่เติบโตจนคล้ายกันก็งอกพ่อแม่ร่วมกันขึ้นมา และเมื่อ inheritance กลายเป็นเครื่องมือที่ผิด — เมื่อ subclass ใช้พ่อแม่เพียงเสี้ยวเดียว หรือความสัมพันธ์แบบ "is-a" เป็นเรื่องโกหก — การเคลื่อนไหวเหล่านี้ช่วยให้คุณสลับมันไปเป็น composition โดยไม่เปลี่ยนสิ่งที่โปรแกรมทำ

## กลิ่นไม่ดี

inheritance ทรงพลังและล้ำเส้นได้ง่าย เมท็อดที่เหมือนกันถูกคัดลอกวางลงไปใน subclass สามตัว ฟิลด์อยู่ใน superclass แต่ subclass เพียงครึ่งหนึ่งเท่านั้นที่แตะมัน คลาสหนึ่ง extends อีกคลาสเพียงเพื่อยืม helper สองตัว แล้วก็ inherit อีกสิบสองตัวที่ต้องแกล้งทำเป็นไม่มี แต่ละกรณีคือลำดับชั้นที่ไม่ได้บอกความจริงเกี่ยวกับวิธีที่ type สัมพันธ์กันอีกต่อไป refactoring ที่อยู่ในนี้จัดลำดับชั้นให้ตรง — หรือสลายมันไปเป็น delegation เมื่อ composition เหมาะกว่า

## โมดูลนี้ครอบคลุมอะไรบ้าง

refactoring แปดตัว แต่ละตัวย้อนกลับได้:

- **Pull Up Method / Field** — เมื่อพี่น้องมีสมาชิกที่เหมือนกัน ให้ย้ายมันขึ้นไปไว้ที่ superclass เพื่อให้มันอยู่ในที่เดียวเป๊ะ ๆ ส่วน **Push Down** เป็นสิ่งตรงข้าม: สมาชิกที่มีเพียงบาง subclass ต้องใช้จะหล่นลงไปอยู่ใน subclass เหล่านั้น
- **Extract Superclass / Interface** — เมื่อสองคลาสมีฟีเจอร์ร่วมกัน ให้ยกความเหมือนขึ้นไปเป็นพ่อแม่ใหม่ และเมื่อมันมีเพียง protocol ร่วมกัน ให้ extract เป็น interface (เป็น trait ใน Rust)
- **Replace Inheritance with Delegation** — เมื่อ subclass พึ่งพาพ่อแม่เพียงบางส่วน ให้ถือพ่อแม่เดิมไว้เป็นฟิลด์แล้ว delegate ไปหามัน
- **Replace Subclass with Delegation** — เมื่อ subclass แตกต่างกันตามแกนเดียว ให้แทนลำดับชั้นทั้งหมดด้วยคลาสเดียวที่ถือ strategy object ไว้
- **Collapse Hierarchy** — เมื่อ subclass กับ superclass ของมันใกล้กันเกินกว่าจะให้มีสอง type ได้ ก็รวมมันเป็นหนึ่ง
- **[Replace Subclass with Fields](/refactoring-from-zero-to-hero/th/generalization-and-inheritance/replace-subclass-with-fields/)** — เมื่อ subclass ต่างกันเพียงข้อมูลคงที่ที่ถูกคืนกลับจากเมท็อด ให้ยุบมันเป็นคลาสเดียวที่มีฟิลด์ถือข้อมูลนั้นไว้
- **[Replace Superclass with Delegation](/refactoring-from-zero-to-hero/th/generalization-and-inheritance/replace-superclass-with-delegation/)** — เมื่อ subclass ใช้ superclass เพียงบางส่วน หรือ inherit interface ที่เก้งก้าง ให้ถือ superclass เดิมไว้เป็นฟิลด์แล้ว delegate ไปหามัน
- **[Pull Up Constructor Body](/refactoring-from-zero-to-hero/th/generalization-and-inheritance/pull-up-constructor-body/)** — เมื่อ constructor ของพี่น้องมีการ initialize เหมือนกัน ให้ย้ายการสร้างที่เหมือนกันนั้นเข้าไปไว้ใน constructor ของ superclass แล้วเรียกขึ้นไปหามัน

แต่ละบทเรียนแสดง before-and-after เดียวกันใน TypeScript, Python, Go และ Rust สังเกตว่า **Go และ Rust ไม่มี class inheritance**: Go ประกอบพฤติกรรมผ่าน *struct embedding* และ interface ส่วน Rust ผ่าน *trait* และฟิลด์ที่ถือไว้ ดังนั้นตัวอย่างในภาษาเหล่านั้นจึงจำลองเจตนาเดียวกันด้วย composition — ซึ่งอย่างที่คุณจะได้เห็น คือจุดที่ refactoring หลายตัวเหล่านี้พยายามนำคุณไปอยู่แล้ว

<Callout title="inheritance เป็นเครื่องมือคม ไม่ใช่ค่าตั้งต้น">
หยิบลำดับชั้นมาใช้เมื่อ subtype *เป็น* ชนิดหนึ่งของ supertype จริง ๆ และใช้ interface ทั้งหมดของมันร่วมกัน ทันทีที่ subclass เริ่ม override สมาชิกให้กลายเป็น no-op หรือยืมพ่อแม่เพียงเสี้ยวเดียว ความสัมพันธ์นั้นกำลังโกหก — และ refactoring แบบ delegation ในโมดูลนี้คือทางออกของคุณ Go และ Rust ทำให้บทเรียนนี้เป็นเชิงโครงสร้าง: เมื่อไม่มี inheritance ให้พึ่ง คุณก็ประกอบด้วย composition โดยปริยาย
</Callout>
