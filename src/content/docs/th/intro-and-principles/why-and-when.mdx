## ทำไมต้องเสียเวลาด้วย?

Refactoring ไม่ได้เพิ่มฟีเจอร์และไม่ได้ทำให้ผู้ใช้พึงพอใจโดยตรง จึงเป็นธรรมที่จะถามว่าคุณได้อะไรกลับมาจากความพยายามนี้ สามอย่าง ซึ่งทั้งหมดทบต้นขึ้นตลอดอายุของโปรเจกต์

- **อ่านง่าย** เวลาส่วนใหญ่ของนักพัฒนาหมดไปกับการอ่าน code ไม่ใช่การเขียน code ที่ปรับรูปทรงให้เผยเจตนาของมันถูกอ่านได้เร็วกว่าและเข้าใจผิดน้อยกว่า ทุกนาทีที่ใช้ไปกับการทำให้ code ชัดเจน ถูกตอบแทนกลับมาหลายเท่าโดยทุกคนที่อ่านมันในภายหลัง — รวมถึงตัวคุณเองในเดือนหน้าด้วย
- **บั๊กน้อยลง** โครงสร้างที่ชัดเจนเผยให้เห็นข้อบกพร่องที่ code ยุ่งเหยิงซ่อนไว้ เมื่อแต่ละ function ทำสิ่งเดียวที่ตั้งชื่อได้ สมมติฐานที่ผิดก็ไม่มีที่ให้ซุกซ่อน Refactoring ไม่ได้แก้บั๊กด้วยตัวมันเอง แต่มันทำให้บั๊กมองเห็นได้ — และทำให้การแก้ในที่สุดนั้นปลอดภัย
- **เปลี่ยนแปลงถูกลง** ผลตอบแทนที่แท้จริง code ที่มีโครงสร้างดีต่อยอดได้ถูก ส่วน code ที่ยุ่งเหยิงทำให้ทุกฟีเจอร์ใหม่กลายเป็นโครงการขุดค้นทางโบราณคดี Refactoring คือการลงทุนที่ทำให้ต้นทุนของการเปลี่ยนแปลงในอนาคตยังคงต่ำ

## Technical debt

วิธีที่มีประโยชน์ในการคิดถึง code ที่รกรุงรังคือ **technical debt** เมื่อคุณใช้ทางลัดเพื่อส่งงานให้เร็วขึ้น คุณกำลังกู้ยืมจากอนาคต code ทำงานได้ตอนนี้ แต่มันแก้ยากขึ้น เหมือนหนี้ทางการเงิน มันคิด *ดอกเบี้ย* — การเปลี่ยนแปลงในอนาคตทุกครั้งในบริเวณนั้นจะแพงกว่าที่ควรเป็นเล็กน้อย

เงินกู้ก้อนเล็กที่ตั้งใจกู้อาจเป็นเรื่องฉลาด มันช่วยให้ฟีเจอร์ออกสู่ตลาดได้ อันตรายคือการไม่เคยชำระคืนเลย เมื่อถูกปล่อยปละ ดอกเบี้ยจะทบต้นจนการเปลี่ยนแปลงบรรทัดเดียวกินเวลาเป็นสัปดาห์ Refactoring คือวิธีที่คุณชำระเงินต้นและหยุดดอกเบี้ย

## กฎลูกเสือ

> จงทิ้ง code ไว้ให้สะอาดกว่าตอนที่คุณพบมันเล็กน้อยเสมอ

คุณไม่จำเป็นต้องมี "refactoring sprint" โดยเฉพาะ การปรับปรุงที่ยั่งยืนที่สุดมาจากการเก็บกวาดเล็ก ๆ ที่สอดแทรกอยู่ในงานปกติ เปลี่ยนชื่อตัวแปรที่สับสนหนึ่งตัว extract บล็อกที่ยุ่งเหยิงหนึ่งบล็อก ลบ branch ที่ตายแล้วหนึ่งอันทุกครั้งที่คุณผ่านมา code เบสจะค่อย ๆ มีแนวโน้มสะอาดขึ้นโดยไม่มีใครต้องจัดตารางเวลาเลย

## สามโอกาสที่ควร refactor

ในทางปฏิบัติ refactoring ที่ดีเป็นแบบ *ฉวยโอกาส* — มันแล่นไปพร้อมกับงานที่คุณกำลังทำอยู่แล้ว มีสามจังหวะตามธรรมชาติ

- **Preparatory refactoring** — *"ทำให้การเปลี่ยนแปลงง่าย แล้วค่อยทำการเปลี่ยนแปลงที่ง่ายนั้น"* ก่อนเพิ่มฟีเจอร์ จงปรับรูปทรงของ code โดยรอบเพื่อให้ฟีเจอร์ลงตัวอย่างสะอาด ยังไงคุณก็กำลังจะอ่าน code นี้อยู่แล้ว ใช้เวลาสองสามนาทีทำให้มันพร้อม
- **Comprehension refactoring** — เมื่อคุณเข้าใจ code ที่สับสนชิ้นหนึ่งได้ในที่สุด จงเข้ารหัสความเข้าใจนั้นกลับเข้าไปในมัน เปลี่ยนชื่อสิ่งต่าง ๆ extract ส่วนที่คุณเพิ่งไขออก เพื่อให้ผู้อ่านคนต่อไปไม่ต้องดิ้นรนซ้ำอย่างที่คุณเจอ
- **Litter-pickup refactoring** — คุณสังเกตเห็นความรกรุงรังที่ไม่เกี่ยวข้องขณะผ่านมา ถ้ามันเล็ก ก็แก้ตรงนั้นเลย ถ้ามันใหญ่ ก็จดไว้แล้วกลับมาทีหลัง อย่าปล่อยให้การเก็บกวาดบานปลายเป็น diff ที่รีวิวไม่ไหว

## เมื่อไหร่ที่ *ไม่* ควร refactor

Refactoring ไม่ได้ฟรี และมีบางเวลาที่ควรวางหมวกลง

| สถานการณ์ | ทำไมจึงควรยับยั้งไว้ |
| --- | --- |
| **ไม่มี test** | หากไม่มีตาข่ายความปลอดภัย คุณก็พิสูจน์ไม่ได้ว่า behavior ถูกรักษาไว้ จงเพิ่ม characterization test ก่อน *แล้วจึง* refactor |
| **code ใช้แล้วทิ้ง** | สคริปต์ครั้งเดียวจบหรือ spike ที่คุณจะลบทิ้งพรุ่งนี้ ไม่คุ้มกับการลงทุน |
| **ช่วงคับขันก่อนเส้นตาย** | ถ้าการแก้ต้องส่งให้ได้ในชั่วโมงหน้า ก็ส่งมันไป จดหนี้ไว้และชำระคืนอย่างตั้งใจในภายหลัง — ไม่ใช่ตอนที่อยู่ใต้แรงกดดัน |
| **code ที่คุณกำลังจะลบ** | การปรับโครงสร้าง code ที่กำลังจะถูกนำออกในตั๋วถัดไป เป็นความพยายามที่สูญเปล่า |

กฎลูกเสือฉบับซื่อสัตย์ย่อมรวมถึงการรู้ว่าเมื่อไหร่ที่ที่ตั้งแคมป์กำลังไฟไหม้และตอนนี้ไม่ใช่ช่วงเวลาที่จะเก็บกวาด
