## pattern มาจากไหน

ไอเดียของการจดบันทึกวิธีแก้ปัญหาที่เกิดขึ้นซ้ำ ๆ ไม่ได้เริ่มต้นจากซอฟต์แวร์ มันมาจากสถาปนิกคนหนึ่งชื่อ Christopher Alexander ผู้ซึ่งรวบรวมการจัดวางที่นำกลับมาใช้ซ้ำได้สำหรับอาคารและเมือง ในช่วงกลางทศวรรษ 1990 ผู้เขียนสี่คน — ที่รวมกันได้ฉายาว่า Gang of Four — ได้ยืมแนวทางนั้นมาใช้กับซอฟต์แวร์เชิงวัตถุ และตีพิมพ์แคตตาล็อกของ pattern ยี่สิบสามแบบ คุณูปการที่ยั่งยืนของพวกเขาไม่ได้อยู่ที่ pattern เฉพาะเจาะจงเท่ากับ*รูปแบบ*: วิธีร่วมกันในการตั้งชื่อปัญหา อธิบายโครงสร้างที่พิสูจน์แล้ว และชั่งน้ำหนักผลที่ตามมา เมื่อนักพัฒนามีชื่อเรียกร่วมกัน บทสนทนาเรื่องการออกแบบก็สั้นลงอย่างมาก

มันคุ้มค่าที่จะจำไว้ว่า pattern ถูก*ค้นพบ* ไม่ได้ถูก*ประดิษฐ์ขึ้น* ผู้เขียนสำรวจระบบที่ใช้งานได้ดีอยู่แล้วและสังเกตเห็นรูปทรงเดียวกันเกิดขึ้นซ้ำ ๆ ต้นกำเนิดนั้นอธิบายทั้งจุดแข็งและข้อจำกัดของมัน: pattern เก็บรวบรวมประสบการณ์ที่ได้มาอย่างยากลำบาก แต่เฉพาะกับชนิดของปัญหาที่ผู้คนเคยแก้กันมาจริง ๆ เท่านั้น

## ประโยชน์

- **คำศัพท์ร่วมกัน** ทีมที่รู้จักแคตตาล็อกสามารถอธิบายการออกแบบที่ไม่ธรรมดาได้ในประโยคเดียว และเชื่อมั่นได้ว่าทุกคนนึกภาพสิ่งเดียวกัน
- **โครงสร้างที่พิสูจน์แล้ว** แต่ละ pattern ผ่านการทดสอบความทนทานมาแล้วในหลายระบบ ดังนั้นคุณจึงได้รับวิธีแก้ปัญหาที่มีการบันทึกข้อพึงระวังไว้แล้วเป็นมรดก
- **การให้เหตุผลที่เร็วขึ้น** การจดจำได้ว่า "นี่คือปัญหาแบบ Observer" ช่วยให้คุณข้ามการค้นพบการจัดวางนั้นใหม่ และกระโดดไปยังการปรับใช้มันได้เลย

## คำวิจารณ์

pattern ไม่ได้มาฟรี ๆ และการปฏิบัติกับมันเป็นเป้าหมายแทนที่จะเป็นเครื่องมือก่อให้เกิดอันตรายจริง ๆ

- **Over-engineering** ความล้มเหลวที่พบบ่อยที่สุดคือการเอื้อมไปหยิบ pattern ก่อนที่ปัญหาจะสมเหตุสมผลให้ทำ — ห่อความต้องการบรรทัดเดียวด้วย interface สามอันกับ factory การเพิ่มชั้นทางอ้อมที่คุณไม่ต้องการก็เป็นแค่ต้นทุน
- **การแก้ขัดสำหรับฟีเจอร์ที่ขาดหายไป** pattern คลาสสิกบางตัวมีอยู่เพื่อปะรอยช่องว่างของภาษาเป็นหลัก ตัวอย่างเช่น first-class function ทำให้ "behavioral" pattern หลายตัวหดเหลือเพียง callback อันเดียว
- **Cargo-culting** การนำ pattern ไปใช้เพราะมันดูมีเกียรติ มากกว่าเพราะแรงผลักดันตรงกัน ก่อให้เกิดพิธีรีตองโดยไม่มีประโยชน์

## โครงสร้างของ pattern

ทุก pattern ในคอร์สนี้ — และในแคตตาล็อกต้นฉบับ — ถูกอธิบายด้วยชุดฟิลด์เพียงไม่กี่อย่างที่เหมือนกัน การอ่านมันไปตามลำดับจะบอกคุณว่า pattern เหมาะสมหรือไม่ก่อนที่คุณจะเขียนโค้ดสักบรรทัด

- **Intent** — ประโยคเดียวที่บอกว่า pattern ทำอะไรให้สำเร็จ
- **Problem** — สถานการณ์ที่เกิดขึ้นซ้ำ ๆ และแรงผลักดันที่ขัดแย้งกัน (ความยืดหยุ่น การ coupling ประสิทธิภาพ) ที่ทำให้มันยาก
- **Structure** — บทบาทที่เข้าร่วมและความสัมพันธ์ของพวกมัน มักแสดงเป็น UML class diagram
- **Consequences** — สิ่งที่คุณได้และสิ่งที่คุณจ่าย: ข้อแลกเปลี่ยนที่ตัดสินว่า pattern นั้นคุ้มค่าหรือไม่ในกรณีนี้

## เมื่อไรควรเอื้อมไปหยิบมา

จงเอื้อมไปหยิบ pattern เมื่อคุณรู้สึกถึงความเจ็บปวดที่เฉพาะเจาะจงและเกิดขึ้นซ้ำ ๆ — ตรรกะการสร้างที่ซ้ำซ้อน บันได `if`/`switch` ที่แข็งทื่อบน type หนึ่ง หรือ class ที่รู้เรื่องของอีก class มากเกินไป ความเจ็บปวดมาก่อน pattern คือการตอบสนอง ถ้าคุณไม่สามารถระบุชื่อแรงผลักดันที่ pattern หนึ่งช่วยบรรเทาได้ แสดงว่าคุณยังไม่ต้องการมัน เครื่องหมายของนักออกแบบที่มีประสบการณ์ไม่ได้อยู่ที่ว่าพวกเขาใช้ pattern มากแค่ไหน แต่อยู่ที่ว่าพวกเขาตัดสินใจ*ไม่*ใช้ได้อย่างถูกต้องบ่อยแค่ไหน

<Callout title="ปล่อยให้ปัญหาเรียกร้องหา pattern">
pattern จะคุ้มค่ากับสิ่งที่เสียไปก็ต่อเมื่อแรงผลักดันที่มันสร้างสมดุลให้นั้นมีอยู่จริง จงเริ่มต้นด้วยสิ่งที่ง่ายที่สุดที่ใช้งานได้ แล้วค่อยนำ pattern เข้ามาเมื่อความซ้ำซ้อน ความแข็งทื่อ หรือการ coupling ทำให้โค้ดง่าย ๆ นั้นเจ็บปวด — อย่าก่อนหน้านั้น
</Callout>
