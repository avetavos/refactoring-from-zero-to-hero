## เจตนา

ตระกูล **simplifying conditionals** ว่าด้วยเรื่องตรรกะการตัดสินใจ ได้แก่ สาขา (branch), การ์ด (guard), และการสลับ (switch) ที่ตัดสินว่า *โปรแกรมจะทำอะไรต่อไป* เงื่อนไขที่เริ่มต้นจากคำถามเดียวอย่างชัดเจน ค่อย ๆ งอกอนุประโยคเพิ่มเข้ามา ซ้อนตัวเองอยู่ภายในเงื่อนไขอื่น ๆ และแตกตัวเป็นสำเนากระจายไปทั่วฐานโค้ด การ refactor เหล่านี้ช่วยคลายความซับซ้อนที่งอกออกมา เพื่อให้ผู้อ่านมองเห็นได้ในพริบตาว่ากรณีใดนำไปสู่จุดไหน

## กลิ่นโค้ด

ตรรกะเงื่อนไขที่ซับซ้อนเป็นหนึ่งในสัญญาณที่ดังที่สุดว่าฟังก์ชันหนึ่งดูแลรักษายาก สัญญาณเตือนนั้นคุ้นเคยกันดี ได้แก่ นิพจน์บูลีนที่ยาวจนต้องอ่านสองรอบ, แถวของบล็อก `if` ที่ซ้อนกันจนพุ่งทะลุขอบขวาของหน้าจอ, `switch` บนฟิลด์ "ชนิด" ที่ซ้ำกันอยู่ในห้าฟังก์ชัน, และ `if (x == null)` ที่กระจัดกระจายอยู่ในทุกเมธอดที่แตะต้อง `x` แต่ละอย่างมีวิธีแก้ที่มีชื่อเรียกชัดเจน และแต่ละวิธีก็ทิ้งพฤติกรรมไว้เหมือนเดิมพร้อมกับทำให้เจตนาชัดเจนขึ้น

## โมดูลนี้ครอบคลุมอะไรบ้าง

การ refactor ห้าแบบ แต่ละแบบเล็กและย้อนกลับได้:

- **Decompose Conditional** — แยกการทดสอบ, สาขา then, และสาขา else ออกมาเป็นฟังก์ชันที่ตั้งชื่ออย่างดี เพื่อให้โครงสร้างอ่านได้เหมือนประโยค: *ถ้าเป็นวันพีค ให้คิดอัตราพีค มิฉะนั้นคิดอัตราปกติ*
- **Consolidate Conditional Expression** — เมื่อการตรวจสอบหลายตัวที่แยกกันนำไปสู่ผลลัพธ์เดียวกัน ให้รวมมันเป็นเงื่อนไขเดียวและตั้งชื่อ เพื่อให้ผู้อ่านเรียนรู้ว่า *ทำไม* แทนที่จะเป็นว่า *มีกี่วิธี*
- **Replace Nested Conditional with Guard Clauses** — ทำให้ `if/else` ที่ซ้อนลึกแบนราบลง โดย return ออกไปก่อนสำหรับกรณีพิเศษ เหลือเส้นทางหลักไว้แบบไม่ย่อหน้าและเห็นได้ชัด
- **Replace Conditional with Polymorphism** — เมื่อ `switch` หรือ `if` บนชนิดข้อมูลเกิดซ้ำ ให้ผลักแต่ละสาขาเข้าไปเป็นเมธอดของซับคลาส เพื่อให้ภาษากระจายงาน (dispatch) ให้คุณ
- **Introduce Special Case** — แทนการตรวจสอบ null-หรือ-พิเศษ ที่ซ้ำ ๆ ด้วยอ็อบเจกต์ที่ตอบกลับด้วยค่าดีฟอลต์ที่สมเหตุสมผล เพื่อให้ผู้เรียกหยุดถามว่า "มันหายไปไหม?"

แต่ละบทเรียนแสดง before-and-after เดียวกันใน TypeScript, Python, Go, และ Rust จากนั้นพาเดินผ่านกลไกที่ปลอดภัยและเรียงลำดับ — โดยรันเทสต์ของคุณระหว่างทุกขั้นตอน
