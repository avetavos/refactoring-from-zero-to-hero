## Intent

code ชิ้นหนึ่งมักทำงานได้ก็ต่อเมื่อบางอย่างเป็นจริงอยู่ก่อนแล้ว — ส่วนลดอยู่ระหว่าง 0 ถึง 1, ลิสต์ไม่ว่างเปล่า, บัญชีถูกเปิดแล้ว ข้อสมมตินั้นมักอยู่ในหัวของนักพัฒนา ไม่ใช่ใน code **Introduce Assertion** เขียนข้อสมมตินั้นออกมาเป็นการตรวจสอบที่ไม่ทำอะไรเลยเมื่อโลกเป็นอย่างที่คาดหวัง และหยุดโปรแกรมอย่างดังเมื่อมันไม่เป็นเช่นนั้น

assertion คือคำกล่าวเกี่ยวกับสิ่งที่ *ต้อง* เป็นจริง ณ จุดหนึ่งใน code มันไม่ใช่การจัดการ error สำหรับอินพุตที่คุณคาดว่าจะผิด แต่เป็นสายสะดุดสำหรับสถานการณ์ที่ควรเป็นไปไม่ได้ ถ้ามันถูกจุดชนวนเมื่อใด แสดงว่าผู้เรียกมีบั๊ก

## Code Smell

คุณอ่านการคำนวณหนึ่งแล้วคิดว่า *"อันนี้จะสมเหตุสมผลก็ต่อเมื่อเรตเป็นบวกเท่านั้น"* — แต่ไม่มีอะไรใน code บอกไว้เลย ข้อสมมตินั้นมองไม่เห็น ดังนั้นวันที่ศูนย์หรือค่าลบเล็ดลอดเข้ามา function ก็ผลิตขยะออกมาอย่างมีความสุข ความเสียหายโผล่ขึ้นมาห่างไปสามชั้น ในยอดรวมที่ผิดอย่างเงียบ ๆ และคุณก็เสียเวลาทั้งบ่ายไล่ตามมันย้อนกลับมา เงื่อนไขที่ code พึ่งพาไม่เคยถูกกล่าวไว้ จึงไม่มีอะไรเฝ้าระวังมัน

## Before → After

function หนึ่งใช้ส่วนลดที่ต้องเป็นเศษส่วนระหว่าง 0 ถึง 1 ก่อนหน้านี้ ข้อจำกัดนั้นไม่ได้เขียนไว้ หลังจากนั้น assertion กล่าวมันออกมา เพื่อให้เรตที่ไม่ดีล้มเหลวตั้งแต่ที่ประตู

<Tabs syncKey="lang">
  <TabItem label="TypeScript">
    ```ts
    // Before
    function applyDiscount(price: number, rate: number): number {

    function applyDiscount(price: number, rate: number): number {

    }
    ```

  <TabItem label="Python">
    ```python
    # Before
    def apply_discount(price, rate):
        # silently assumes 0 <= rate <= 1
        return price - price * rate

    # After
    def apply_discount(price, rate):
        assert 0 <= rate <= 1, f"discount rate must be in [0, 1], got {rate}"
        return price - price * rate
    ```
  </TabItem>
  <TabItem label="Go">
    ```go
    // Go has no assert keyword. State the invariant with an explicit
    // check and panic for a true "this can never happen" bug. If the
    // bad value can come from outside, return an error instead.
    //
    // Before
    func ApplyDiscount(price, rate float64) float64 {
        // silently assumes 0 <= rate <= 1
        return price - price*rate
    }

    // After
    func ApplyDiscount(price, rate float64) float64 {
        if rate < 0 || rate > 1 {
            panic(fmt.Sprintf("discount rate must be in [0, 1], got %v", rate))
        }
        return price - price*rate
    }
    ```
  </TabItem>
  <TabItem label="Rust">
    ```rust
    // Use assert! for an invariant that must hold in every build, or
    // debug_assert! when the check is only worth paying for in dev builds.
    //
    // Before
    fn apply_discount(price: f64, rate: f64) -> f64 {
        // silently assumes 0.0 <= rate <= 1.0
        price - price * rate
    }

    // After
    fn apply_discount(price: f64, rate: f64) -> f64 {
        assert!(
            (0.0..=1.0).contains(&rate),
            "discount rate must be in [0, 1], got {rate}"
        );
        price - price * rate
    }
    ```
  </TabItem>

## Mechanics

1. หาข้อสมมติให้เจอ อ่าน code ที่น่าสงสัยแล้วเติมประโยค *"อันนี้ทำงานได้ก็ต่อเมื่อ…"* ให้จบ อนุประโยคนั้นคือ assertion ของคุณ
2. แสดงมันออกมาเป็นการตรวจสอบที่เป็นจริงในกรณีปกติ วางมันไว้ใกล้กับ code ที่พึ่งพามันให้มากที่สุด ก่อนที่งานจะเริ่ม
3. ทำให้ความล้มเหลวดังและเฉพาะเจาะจง: ใส่ค่าที่ผิดลงในข้อความ เพื่อให้ผู้อ่านในอนาคตเห็นว่า *อะไร* ที่ทำลายข้อสมมติ ไม่ใช่แค่ *ว่า* มันพัง
4. **รันเทสต์ของคุณ** เส้นทางที่ปกติดีต้องผ่านต่อไปโดยไม่ถูกแตะต้อง — assertion ที่ถูกต้องไม่เปลี่ยนแปลงอะไรสำหรับอินพุตที่ถูกต้อง
5. เพิ่มเทสต์ที่ป้อนค่าที่ผิดกฎเข้าไป แล้วยืนยันว่า assertion ถูกจุดชนวน นี่พิสูจน์ว่าสายสะดุดถูกต่อไว้แล้ว
6. ทำซ้ำสำหรับข้อสมมติถัดไปที่ไม่ได้กล่าวไว้ ต้านทานการ assert สิ่งที่ชัดเจนว่าถูกรับประกันโดยชนิดข้อมูลอยู่แล้ว ให้ assert สิ่งที่ *ผู้เรียก* อาจทำผิดได้จริง

## When to use / trade-offs

introduce assertion เมื่อ code ส่วนหนึ่งพึ่งพาเงื่อนไขอยู่เงียบ ๆ และการละเมิดมันจะทำให้ข้อมูลเสียหายในที่ห่างไกลจากต้นเหตุ assertion ทำหน้าที่เป็นเอกสารไปด้วย: มันบอกผู้อ่านคนต่อไปอย่างชัดเจนว่า code คาดหวังอะไร และต่างจากคอมเมนต์ตรงที่มันไม่สามารถเคลื่อนคลาดออกจากความจริงได้ เพราะอันที่เป็นเท็จจะล้มเหลว

เส้นที่ต้องเฝ้าระวังคือ **assertion เทียบกับการจัดการ error** assertion เฝ้าระวังความผิดพลาดของ *นักพัฒนา* — สถานะที่ควรเป็นไปไม่ได้ถ้าผู้เรียกทุกคนถูกต้อง อินพุตที่อาจผิดรูปได้อย่างชอบธรรม (ผู้ใช้พิมพ์ผิด, payload เครือข่ายที่ไม่ดี) ไม่ใช่หน้าที่ของ assertion ให้ตรวจสอบความถูกต้องของมันแล้ว return error จริงออกไป ใน Rust นี่คือทางเลือกระหว่าง `assert!`/`debug_assert!` กับ `Result` ใน Go คือ `panic` เทียบกับ `error` ที่ return ออกมา อย่าใส่ side effect ไว้ใน 

 assertion เพราะ build ที่ถอด assertion ออก (Python's `-O`, Rust's `debug_assert!`) จะเปลี่ยน behavior อย่างเงียบ ๆ

| ใช้ Introduce Assertion เมื่อ | หลีกเลี่ยงเมื่อ |
|---|---|
| code พึ่งพาเงื่อนไขโดยไม่กล่าวไว้ชัดเจน | input นั้นมาจาก user หรือ network — ใช้ error handling แทน |
| ต้องการให้ violation หยุดที่ต้นตอ ไม่ใช่ propagate | assertion ที่จะ trigger ใน production flow ปกติ |
| ข้อสมมติที่ละเมิดแล้วสร้างความเสียหายที่ห่างไกล | assertion มี side effect — จะหายไปใน optimized build |

> ⚠️ ไม่ควร Introduce Assertion เมื่อ:
> - เงื่อนไขนั้น validate input จาก user หรือ external system — ควร return error แทน
> - assertion จะ trigger ใน normal flow บ่อย ๆ — แสดงว่ามันคือ expected condition
> - ใส่ side effect ใน assertion — behavior จะเปลี่ยนเมื่อ assertion ถูกปิด
