## เจตนา

เอาค่าที่บางครั้งหายไป — และการตรวจสอบ null ที่ซ้ำอยู่ทุกที่ที่ค่านั้นถูกใช้ — แล้วแทนกรณีที่หายไปด้วยอ็อบเจกต์จริงที่รู้คำตอบดีฟอลต์ที่ถูกต้อง กรณีพิเศษ (**special case**) นี้ (มักเป็น **null object**) ทำตามอินเทอร์เฟซเดียวกัน ดังนั้นผู้เรียกก็แค่ใช้มัน สาขา "จะเป็นอย่างไรถ้ามันไม่มี?" หายไปจากทุกจุดที่เรียก และไปอยู่ในคลาสเดียวแทน

## กลิ่นโค้ด

นี่คือวิธีแก้สำหรับ **การตรวจสอบกรณีพิเศษที่ซ้ำ ๆ** ซึ่งโดยทั่วไปคือ `if (x == null)` เมื่อฟิลด์หนึ่งสามารถหายไปได้ ทุกเมธอดที่แตะต้องมันก็งอกการ์ดตัวเดียวกัน แต่ละตัวแทนค่าดีฟอลต์ของตัวเอง — ชื่อตรงนี้ แพลนตรงนั้น จำนวนเงินค่าบริการที่อื่น ความรู้ว่า "ลูกค้าที่หายไปหมายความว่าอะไร" เลอะเทอะกระจายไปทั่วฐานโค้ด และดีฟอลต์ใหม่ใด ๆ ก็ต้องเพิ่มในที่อีกแห่งหนึ่ง อ็อบเจกต์ special-case รวบรวมค่าดีฟอลต์ทั้งหมดเหล่านั้นไว้ในชนิดเดียวที่ตอบกลับตามปกติ

## Before → After

โค้ดแสดงผลอ่านชื่อและแพลนของลูกค้า โดยใช้ค่าดีฟอลต์สำหรับลูกค้าที่ไม่รู้จัก ก่อนหน้านี้ ทุกจุดตรวจสอบ null หลังจากนั้น `UnknownCustomer` ให้ค่าดีฟอลต์มา

## กลไก

1. เพิ่มพร็อพเพอร์ตีหรือเมธอดที่ให้ผู้เรียกถามได้ว่าค่าหนึ่งเป็นกรณีพิเศษหรือไม่ (ตัวอย่างเช่น `isUnknown`) และทำให้อ็อบเจกต์ปกติตอบ `false`
2. สร้างอ็อบเจกต์ special-case — ซับคลาส, struct, หรือ factory — ที่ทำตามอินเทอร์เฟซเดียวกันและ return ค่าดีฟอลต์ที่ตกลงกันไว้สำหรับแต่ละ accessor
3. จัดให้แหล่งที่มาของค่าส่งอ็อบเจกต์ special-case กลับมาแทน null เมื่อค่านั้นหายไป
4. แทนแต่ละจุด `if (x == null)` ด้วยการใช้ค่านั้นโดยตรง ทีละจุด **รันเทสต์ของคุณ** หลังแต่ละครั้ง
5. เมื่อการตรวจสอบที่ชัดแจ้งตัวสุดท้ายหายไป ให้ลบโค้ดจัดการ null ที่ตายไปแล้วทิ้ง

## เมื่อไรควรใช้ / ข้อแลกเปลี่ยน

หยิบ Introduce Special Case มาใช้เมื่อการตรวจสอบค่าที่หายไป *ตัวเดียวกัน* ซึ่งมีดีฟอลต์ *ตัวเดียวกัน* เกิดซ้ำข้ามหลายจุดที่เรียก การรวมศูนย์ค่าดีฟอลต์เหล่านั้นไว้ในอ็อบเจกต์เดียวขจัดความซ้ำซ้อน และให้ชื่อกับที่อยู่แก่กรณีที่หายไป

มันไม่ใช่เครื่องมือที่ถูกต้องเสมอ หากค่าที่หายไปหมายความถึงสิ่งที่ต่างกันในที่ต่างกัน — ดีฟอลต์ตรงนี้ ข้อผิดพลาดตรงนั้น — อ็อบเจกต์ special-case ตัวเดียวก็ไม่สามารถรองรับได้ทั้งหมด และการบังคับให้ใช้ตัวเดียวจะซ่อนความแตกต่างที่สำคัญ สำหรับค่าที่ถูกตรวจสอบเพียงหนึ่งหรือสองจุด guard clause ธรรมดาง่ายกว่า ใช้ special case เมื่อ *ความสม่ำเสมอ* ของดีฟอลต์คือสิ่งที่ทำให้มันคุ้มค่าที่จะเป็นชนิดข้อมูล
