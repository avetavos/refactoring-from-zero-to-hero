## Intent

Behavioral patterns อธิบายว่าอ็อบเจ็กต์คุยกันอย่างไร ใครถือความรับผิดชอบใด คำขอเดินทางจากผู้ส่งไปยังผู้จัดการอย่างไร และผู้ร่วมงานคงการประสานกันไว้อย่างไรในขณะที่โปรแกรมทำงาน ที่ creational patterns ตอบว่า *อ็อบเจ็กต์ถือกำเนิดขึ้นมาอย่างไร* และ structural patterns ตอบว่า *อ็อบเจ็กต์ประกอบกันอย่างไร* นั้น behavioral patterns ตอบว่า *อ็อบเจ็กต์ประพฤติร่วมกันอย่างไรเมื่อเวลาผ่านไป*

## Problem

ระบบที่ใช้งานได้จริงนั้นแทบไม่เคยเป็นอ็อบเจ็กต์เดียวที่ทำทุกอย่าง แต่เป็นอ็อบเจ็กต์เล็ก ๆ จำนวนมากที่ส่งข้อความหากัน ตอบสนองต่อการเปลี่ยนแปลง และแบ่งงานกันทำ หากปล่อยไว้โดยไม่จัดการ การโต้ตอบนั้นจะแข็งตัวกลายเป็น dependency ที่พันกันยุ่ง คลาสที่รู้ชนิดที่เป็นรูปธรรมของทุกสิ่งที่มันคุยด้วย เงื่อนไขขนาดยักษ์ที่แตกแขนงตามฟิลด์ status หรืออัลกอริทึมที่เชื่อมติดกับผู้เรียกอย่างแน่นหนาจนไม่อาจสลับได้

Behavioral patterns คลายการเชื่อมโยงเหล่านั้นให้หลวมลง แต่ละ pattern แยกแกนของความผันแปรหนึ่งแกนออกมา ไม่ว่าจะเป็นอัลกอริทึม ผู้รับการแจ้งเตือน ตัวคำขอเอง โหมดปัจจุบันของอ็อบเจ็กต์ ลำดับของขั้นตอน หรือวิธีที่คอลเล็กชันถูกท่องผ่าน แล้วมอบบ้านของมันเองให้ เป้าหมายร่วมคือยอมให้ส่วนหนึ่งของการโต้ตอบเปลี่ยนแปลงได้โดยไม่บังคับให้ส่วนอื่นทุกส่วนต้องเปลี่ยนตามไปด้วย

ภาพรวมของหก pattern โดยสังเขป:

- **Strategy** — รวบรวมตระกูลของอัลกอริทึมที่สับเปลี่ยนกันได้ไว้หลัง interface เดียว เพื่อให้การเลือกอัลกอริทึมผันแปรได้อย่างเป็นอิสระจากโค้ดที่ใช้มัน
- **Observer** — ยอมให้ subject แจ้งเตือนกลุ่มผู้พึ่งพาที่เปลี่ยนแปลงได้โดยอัตโนมัติเมื่อใดก็ตามที่สถานะของมันเปลี่ยน โดยไม่ต้องรู้ว่าพวกเขาเป็นใคร
- **Command** — เปลี่ยนคำขอให้เป็นอ็อบเจ็กต์ที่ยืนได้ด้วยตัวเอง ซึ่งทำให้คำขอสามารถเข้าคิว บันทึก log และย้อนกลับผ่าน undo ได้
- **State** — ยอมให้อ็อบเจ็กต์เปลี่ยนพฤติกรรมเมื่อสถานะภายในเปลี่ยน จนดูเหมือนสลับคลาสในขณะรันแทนที่จะแตกแขนงตามแฟล็ก
- **Template Method** — ตรึงโครงร่างของอัลกอริทึมไว้ในคลาสฐานในขณะที่ยอมให้คลาสลูก override ขั้นตอนแต่ละขั้นได้
- **Iterator** — ให้การเข้าถึงสมาชิกของคอลเล็กชันตามลำดับโดยไม่เปิดเผยว่าคอลเล็กชันเก็บข้อมูลไว้อย่างไร

## When to use / trade-offs

- **ข้อดี:** แต่ละ pattern แยกความเปลี่ยนแปลงชนิดหนึ่งออกมา พฤติกรรมจึงวิวัฒน์ได้โดยไม่กระเพื่อมไปทั่วโค้ดที่ไม่เกี่ยวข้อง
- **ข้อดี:** พวกมันแทนที่เงื่อนไขที่บานปลายและห่วงโซ่การเรียกที่แข็งทื่อด้วยผู้ร่วมงานเล็ก ๆ ที่มีชื่อและทดสอบได้
- **ข้อเสีย:** ทุก pattern เพิ่มอ็อบเจ็กต์และชั้นการอ้อม สำหรับตรรกะเรียบง่ายการเรียกตรง ๆ หรือลูปธรรมดาจะชัดเจนกว่า
- **ข้อเสีย:** การใช้มากเกินไปกระจายพฤติกรรมไปทั่วคลาสจิ๋วจำนวนมาก ซึ่งอาจบดบังภาพการไหลโดยรวมได้
