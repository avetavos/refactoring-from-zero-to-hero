## เจตนา

Facade ให้ interface เดียวที่รวมเป็นหนึ่งเดียวกับชุดของ interface ใน subsystem มอบจุดเข้าใช้งานที่ง่ายตัวเดียวให้ client แทนที่จะบังคับให้พวกเขาต้องจัดการ class ที่ทำงานร่วมกันจำนวนมากด้วยตัวเอง

## ปัญหา

subsystem ที่มีความสามารถมักมีชิ้นส่วนที่เคลื่อนไหวจำนวนมาก ในการแปลงไฟล์วิดีโอคุณอาจต้องใช้ decoder, audio mixer, ตัวเลือก codec, ตัวคำนวณ bitrate และตัว writer แต่ละตัวมีการตั้งค่า กฎการจัดลำดับ และความซับซ้อนเฉพาะของตัวเอง client ที่แค่ต้องการ "เปลี่ยนไฟล์นี้เป็น MP4" ไม่ควรต้องเรียนรู้ทั้งห้า class สร้าง instance ในลำดับที่ถูกต้อง และต่อ output ของพวกมันเข้าด้วยกัน ความรู้นั้นทำให้ภายในของ subsystem รั่วไหลไปยังผู้เรียกใช้ทุกราย และทำให้ subsystem แก้ไขยาก

Facade คือ class ตัวเดียวที่รู้จักท่าเต้นนี้ มันเปิดเผย method ที่เน้นงานและเล็ก คือ `convert(file, format)` และภายในก็สร้างและประสานงาน object ของ subsystem ในลำดับที่ถูกต้อง client ขึ้นต่อ facade เพียงอย่างเดียว class ของ subsystem ยังใช้งานได้เต็มที่สำหรับผู้เรียกใช้ขั้นสูงที่ต้องการมันโดยตรง แต่กรณีทั่วไปกลายเป็นการเรียกครั้งเดียว เมื่อ refactor subsystem คุณก็แก้ที่ facade ตัวเดียว ไม่ใช่ที่จุดเรียกใช้นับร้อย

<Mermaid code={facadeUml} title="facade วางอยู่หน้า class หลายตัวของ subsystem ไว้หลัง method เดียว" />

- **Facade** (`VideoConverter`) — class เดียวที่ client คุยด้วย มันรู้วิธีประสานงาน subsystem
- **Subsystem classes** (`Decoder`, `AudioMixer`, `Encoder`) — ตัวทำงานจริง พวกมันไม่รู้จัก facade เลยและยังสามารถถูกใช้โดยตรงได้
- **Client** — เรียก facade และยังคงถูกแยกออกจากโครงสร้างของ subsystem

## ตัวอย่าง

facade `VideoConverter` เปลี่ยน request บรรทัดเดียวให้เป็นลำดับที่ถูกต้องของการเรียก decode, mix และ encode

  <TabItem label="TypeScript">
    ```ts
    class Decoder {
      decode = (file: string) => `raw(${file})`;
    }
    class AudioMixer {
      mix = (stream: string) => `mixed(${stream})`;
    }
    class Encoder {
      encode = (stream: string, format: string) => `${stream}.${format}`;
    }

    // Facade: one method hides the subsystem choreography.
    class VideoConverter {
      private decoder = new Decoder();
      private mixer = new AudioMixer();
      private encoder = new Encoder();

      convert(file: string, format: string): string {
        const raw = this.decoder.decode(file);
        const mixed = this.mixer.mix(raw);
        return this.encoder.encode(mixed, format);
      }
    }

    const converter = new VideoConverter();
    console.log(converter.convert('clip.avi', 'mp4')); // mixed(raw(clip.avi)).mp4
    ```

    struct Decoder;
    impl Decoder {
        fn decode(&self, file: &str) -> String {
            format!("raw({file})")
        }
    }

    impl AudioMixer {
        fn mix(&self, stream: &str) -> String {
            format!("mixed({stream})")
        }
    }

    struct Encoder;
    impl Encoder {
        fn encode(&self, stream: &str, format: &str) -> String {
            format!("{stream}.{format}")
        }
    }

    // Facade: hides the subsystem behind one method.
    struct VideoConverter {
        decoder: Decoder,
        mixer: AudioMixer,

## เมื่อไรควรใช้ / ข้อแลกเปลี่ยน

- **ข้อดี:** ย่อ subsystem หลาย class ให้เหลือจุดเข้าใช้งานที่ง่ายตัวเดียวสำหรับกรณีทั่วไป
- **ข้อดี:** แยก client ออกจากภายในของ subsystem ดังนั้น subsystem จึงสามารถ refactor ได้อย่างอิสระ
- **ข้อดี:** ผู้เรียกใช้ขั้นสูงยังเข้าถึง subsystem โดยตรงได้ facade เป็นทางเลือก ไม่ใช่กำแพง
- **ข้อเสีย:** facade อาจเลื่อนไหลไปเป็น god object ที่รู้มากเกินไปถ้าคุณเอาแต่กองฟีเจอร์ใส่มัน
- **ข้อเสีย:** มันอาจซ่อนความยืดหยุ่นที่มีประโยชน์ ล่อใจให้ client ยอมรับเส้นทางที่ง่ายในขณะที่ต้องการเส้นทางที่ละเอียด

## Pattern ที่เกี่ยวข้อง

- [Adapter](/design-patterns-from-zero-to-hero/th/structural/adapter/) เปลี่ยน interface ของ object หนึ่งให้ตรงกับตัวที่คาดหวัง ส่วน Facade คิดค้น interface ที่เรียบง่ายตัวใหม่เอี่ยมครอบ object จำนวนมาก
- [Abstract Factory](/design-patterns-from-zero-to-hero/th/creational/abstract-factory/) สามารถอยู่เบื้องหลัง facade เพื่อสร้าง object ของ subsystem ที่ facade ประสานงาน

<Callout title="facade คือความสะดวก ไม่ใช่กรงขัง">
facade ที่ดีทำให้เส้นทางทั่วไปเรียบง่ายขึ้นโดยไม่ปิดผนึก subsystem จงคง class ที่อยู่เบื้องล่างให้เข้าถึงได้เพื่อไม่ให้ผู้ใช้ระดับสูงถูกขวาง และต้านทานการปล่อยให้ facade ดูดซับความรับผิดชอบทุกอย่าง เมื่อมันโตจนเทอะทะ ให้แยกมันออกเป็น facade ที่โฟกัสหลายตัว
</Callout>
