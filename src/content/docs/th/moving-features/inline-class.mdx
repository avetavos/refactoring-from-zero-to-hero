## Intent

นำคลาสที่หดเล็กลงจนแทบไม่เหลืออะไร — ฟิลด์ไม่กี่ตัวและเมธอดหนึ่งหรือสองตัวที่ไม่คุ้มกับการเป็น type แยกต่างหากอีกต่อไป — มารวมเข้ากับคลาสที่ถือมันไว้ พฤติกรรมไม่เปลี่ยนแปลง คุณเพียงแค่นำชั้นที่หยุดคุ้มกับการมีอยู่ออกไป

## The smell

นี่คือสิ่งตรงข้ามของ **Extract Class** และเป็นยารักษาคลาสที่กลายเป็นชั้นกลางที่ไม่จำเป็น สัญญาณบ่งชี้: คลาสที่มีฟิลด์เดียวและเมธอดเล็กน้อย type ที่มีอยู่เพียงเพื่อถูกห่อโดยผู้เรียกเดียว หรือความรับผิดชอบในอดีตที่การ refactor ก่อนหน้าได้ทำให้กลวงเปล่า หากการอ่านโค้ดหมายถึงการกระโดดไปยังคลาสเล็ก ๆ แล้วกลับมาทันที การกระโดดนั้นเป็นภาระล้วน ๆ

## Before → After

คลาส `TrackingInformation` ที่ถือสองฟิลด์และจัดรูปแบบมัน ถูกใช้โดย `Shipment` เดียว ก่อนหน้านี้ ข้อมูลและเมธอดเดียวของมันอยู่ใน type แยกต่างหาก หลังจากนั้น พวกมันพับเข้าไปใน `Shipment`

## Mechanics

1. บนคลาสที่ดูดซับ ให้ประกาศเมธอดสาธารณะของคลาสที่คุณกำลังจะ inline และให้แต่ละตัวเพียงมอบหมาย (delegate) ไปยัง instance ภายในไปก่อน
2. อัปเดตผู้เรียกทุกตัวของคลาสที่กำลังจะถูกลบให้ผ่านคลาสที่ดูดซับแทน
3. **รันเทสต์ของคุณ** เพื่อยืนยันว่าการมอบหมายทำงานเหมือนกันทุกประการ
4. ย้ายฟิลด์และเมธอดของคลาสภายในเข้าไปในคลาสที่ดูดซับทีละตัว โดยใช้ **Move Field** และ **Move Function**
5. แทนที่บอดี้ของเมธอดที่มอบหมายแต่ละตัวด้วยตรรกะจริงที่ตอนนี้อาศัยอยู่ในเครื่องแล้ว
6. **รันเทสต์ของคุณ** หลังการย้ายแต่ละครั้ง เพื่อให้ความล้มเหลวชี้ไปยังขั้นตอนเดียว
7. ลบคลาสที่ตอนนี้ว่างเปล่าเมื่อไม่มีอะไรอ้างอิงมัน

## When to use / trade-offs

Inline คลาสเมื่อมันหดเล็กลงเหลือฟิลด์เดียวที่มีพฤติกรรมเล็กน้อย เมื่อมันมีอยู่เพียงเพื่อห่อผู้เรียกเดียว หรือเมื่อการแยกครั้งก่อนไม่เคยเติบโตเป็นความรับผิดชอบที่แท้จริงและตอนนี้เพียงเพิ่มการอ้อมค้อม

ต้นทุนคือคลาสที่ดูดซับใหญ่ขึ้น ดังนั้นอย่า inline type ที่ยังทำงานจริงและแยกออกได้ — นั่นคือเส้นทางกลับไปสู่ Large Class สิ่งตรงข้ามคือ **Extract Class**: หากคลาสที่รวมแล้วงอกความรับผิดชอบที่สองในภายหลัง ก็แยกมันออกอีกครั้ง
