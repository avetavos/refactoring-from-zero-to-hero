## Intent

เมื่อ code บรรทัดหนึ่งจับคู่กับการเรียก function บางตัวเสมอ — รันก่อนหรือหลังมันทันที ที่ทุก ๆ จุดเรียก — บรรทัดนั้นควรอยู่ *ภายใน* function ย้ายมันเข้าไป ตอนนี้ function เป็นเจ้าของงานทั้งหมดอย่างครบถ้วน และผู้เรียกก็หดเหลือเพียงการเรียกครั้งเดียวที่ทำพลาดไม่ได้

## The smell

คุณพบคำสั่งเตรียมการหรือคำสั่งปิดท้ายอันเดิมถูก copy-paste ไว้เหนือหรือใต้การเรียก function ทุกที่ที่มันปรากฏ บางทีผู้เรียก `renderName` ทุกตัวเขียน opening tag ก่อน หรือผู้เรียก `fetchOrder` ทุกตัว log บรรทัดเดียวกันหลังจากนั้น คำสั่งที่ทำซ้ำนั้นเป็นส่วนหนึ่งของ behavior ที่ผู้เรียกคาดหวัง แต่กลับอยู่นอก function ที่ซึ่งผู้เรียกรายใหม่ทุกรายต้องจำให้เพิ่มมันเข้าไป ลืมสักครั้งเดียวก็ได้บั๊กเงียบ ๆ การทำซ้ำนี้เป็นสัญญาณว่าคำสั่งนั้นอยากจะย้ายเข้าไป

## Before → After

function ที่ปล่อย HTML รูปภาพของบุคคล ผู้เรียกทุกตัวเขียนบรรทัด heading `<p>` อันเดิมก่อนจะเรียกมันทันที heading นั้นจริง ๆ แล้วเป็นส่วนหนึ่งของ "render บล็อกรูปภาพ" เราจึงย้ายมันเข้าไป

## Mechanics

1. ยืนยันว่าคำสั่งที่เป็นตัวเลือกนั้นรันติดกับการเรียกที่ **ทุก** จุด ถ้าแม้เพียงผู้เรียกรายเดียวทำต่างออกไป การย้ายนี้ผิด — ความแตกต่างนั้นเป็นของจริง
2. ถ้าจุดเรียกมี code อื่นอยู่ ให้ใช้ **Extract Function** ก่อนเพื่อแยกเฉพาะการเรียกบวกกับคำสั่งที่ทำซ้ำออกเป็น helper เดียว ตอนนี้คุณมีที่ทำงานเพียงที่เดียว
3. ย้ายคำสั่งที่ทำซ้ำเข้าไปใน function เป้าหมาย — ไว้บนสุดถ้ามันรันก่อนการเรียก ไว้ล่างสุดถ้ามันรันหลังการเรียก
4. **รันเทสต์ของคุณ**
5. ลบคำสั่งที่ตอนนี้ซ้ำซ้อนออกจากผู้เรียกแต่ละราย ทีละราย โดยรันเทสต์หลังการลบแต่ละครั้ง
6. ถ้าคุณสร้าง helper ชั่วคราวในขั้นที่ 2 ให้ inline มันเมื่อผู้เรียกทุกรายสะอาดแล้ว

## When to use / trade-offs

ใช้สิ่งนี้เมื่อคำสั่งหนึ่งเป็นส่วนหนึ่งของความรับผิดชอบของ function อย่างแท้จริง แต่บังเอิญอยู่นอกมัน การพับมันเข้าไปจะกำจัดการทำซ้ำและทำให้ function เป็นหน่วยที่ครบถ้วนและใช้ผิดได้ยาก

อย่าใช้มันเมื่อผู้เรียกแตกต่างกันในเรื่องว่าจะรันคำสั่งหรือไม่ หรือรันอย่างไร — การบังคับขั้นตอนร่วมลงบน function ที่ผู้เรียกบางรายอยากข้าม เป็นเพียงการแลกการทำซ้ำไปกับ flag parameter ซึ่งแย่กว่า การ refactor ที่ตรงข้ามคือ **Move Statements to Callers**: เมื่อ function มัดขั้นตอนที่มีเพียงผู้เรียก *บางราย* ต้องการ ให้ดันมันออกไป

| ใช้ Move Statements into Function เมื่อ | หลีกเลี่ยงเมื่อ |
|---|---|
| statement เดิมซ้ำทุกครั้งก่อนหรือหลัง call | statement นั้นบางครั้งต้องการ แต่บางครั้งไม่ |
| statement นั้นเป็นส่วนหนึ่งของ invariant ของ function | caller ที่ต่างกันต้องการ behavior ที่ต่างกันหลัง call |
| ทุก caller ทำ statement เดิมหลัง call เสมอ | การย้ายเข้าไปจะซ่อน side effect จากผู้อ่าน caller |

> ⚠️ ไม่ควร Move Statements into Function เมื่อ:
> - มี caller บางตัวที่ไม่ต้องการ statement นั้น
> - statement นั้นขึ้นกับ state ที่ function ไม่รู้จัก
> - ต้องการ flexibility ให้ caller เลือก behavior เอง
