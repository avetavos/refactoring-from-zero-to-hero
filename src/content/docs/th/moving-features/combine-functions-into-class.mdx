## Intent

เมื่อมีฟังก์ชันอิสระหลายตัวที่ทำงานบนกลุ่มข้อมูลชุดเดียวกัน — แต่ละตัวรับมันเข้ามาเป็น argument แต่ละตัวคำนวณค่าจากมัน — ให้รวมพวกมันเข้าเป็นคลาส ข้อมูลที่ใช้ร่วมกันกลายเป็น field ของอ็อบเจกต์ ฟังก์ชันกลายเป็น method ของมัน และ data clump ก็เลิกถูกร้อยส่งผ่านทุก ๆ การเรียก

## The smell

คุณเห็นกลุ่มฟังก์ชันที่รับ record ตัวเดียวกันและดึงค่าออกจากมัน: `baseCharge(reading)`, `taxableCharge(reading)`, `calculateBaseCharge(reading)` record เดินทางจากฟังก์ชันหนึ่งไปอีกฟังก์ชันในรูป parameter และฟังก์ชันเหล่านี้ก็รวมเป็นครอบครัวรอบ ๆ มันอย่างชัดเจน — แต่ไม่มีอะไรตั้งชื่อให้ครอบครัวนั้น นี่คือ **Data Clump** ที่มีพฤติกรรมโคจรอยู่รอบ ๆ การห่อข้อมูลไว้ในคลาสจะให้บ้านแก่ครอบครัวนี้ ให้ method เรียกกันเองได้โดยไม่ต้องส่งข้อมูลซ้ำ และสร้างที่ชัดเจนให้ฟังก์ชันที่เกี่ยวข้องตัวต่อไปมาลง

## Before → After

ฟังก์ชันสามตัวที่แต่ละตัวรับ `reading` ของมิเตอร์แล้วคำนวณค่าใช้จ่าย เรารวมพวกมันเข้าเป็นคลาส `Reading` ที่มี field เป็นข้อมูลของ reading

    // Before — data threaded through every function
    function baseRate(month: number): number {
      return month >= 6 && month <= 9 ? 0.12 : 0.1;

      private get baseRate(): number {
        return this.reading.month >= 6 && this.reading.month <= 9 ? 0.12 : 0.1;
      }

      get baseCharge(): number {
        return this.baseRate * this.reading.quantity;
      }

      get taxableCharge(): number {
        return Math.max(0, this.baseCharge - 30);
      }
    }

    # Before — data threaded through every function
    def base_rate(month):

            self.base_rate() * self.reading.quantity
        }

        fn taxable_charge(&self) -> f64 {
            (self.base_charge() - 30.0).max(0.0)
        }
    }

## Mechanics

1. เลือก record ร่วมที่ทุกฟังก์ชันใช้ และทำ **Encapsulate Record** ถ้ามันเป็นโครงสร้างข้อมูลเปล่า ๆ — คุณต้องการอ็อบเจกต์หนึ่งตัวมายึดคลาสไว้
2. สร้างคลาส (หรือ struct + impl) ที่รับ record นั้นใน constructor และเก็บมันไว้เป็น field
3. ย้ายแต่ละฟังก์ชันเข้าคลาสทีละตัว เปลี่ยน parameter ข้อมูลของมันให้เป็นการอ้างถึง field ที่เก็บไว้ แล้วรันเทสต์หลังการย้ายแต่ละครั้ง
4. แทนที่การเข้าถึง parameter เดิม (เช่น `reading.month`) ด้วยการเข้าถึง field บนอ็อบเจกต์
5. เมื่อ method เริ่มเรียกกันเอง ให้ทิ้ง argument ที่ตอนนี้ซ้ำซ้อนที่เคยส่งกันออกไป
6. **รันเทสต์** หลังทุกขั้น แล้วอัปเดต call site เดิมแต่ละจุดให้สร้างอ็อบเจกต์และเรียก method

## When to use / trade-offs

ใช้สิ่งนี้เมื่อชุดของฟังก์ชันเป็นของข้อมูลชิ้นเดียวอย่างชัดเจน และคุณคอยส่งข้อมูลนั้นไปมาระหว่างพวกมัน คลาสให้ชื่อแก่กลุ่มก้อนนี้ ลบการร้อย parameter ที่ซ้ำซาก และเสนอบ้านตามธรรมชาติให้ตรรกะที่เกี่ยวข้องซึ่งคุณจะเพิ่มเข้ามาภายหลัง มันยังเข้าคู่ได้ดีกับ **Split Phase** และ **Extract Function** — เมื่อฟังก์ชันใช้อ็อบเจกต์ร่วมกัน ค่าที่คำนวณได้ก็กลายเป็น query ที่สะอาดบนมัน

ข้อแลกเปลี่ยนคือคุณกำลังนำอ็อบเจกต์เข้ามาในที่ที่เคยมีแต่ฟังก์ชันธรรมดา ซึ่งคุ้มก็ต่อเมื่อฟังก์ชันเหล่านั้นเกาะกลุ่มกันรอบข้อมูลร่วมจริง ๆ ถ้าฟังก์ชันเพียงบังเอิญรับ argument คล้ายกันแต่มุ่งเป้าที่ไม่เกี่ยวกัน คลาสก็จะจับสิ่งที่ไม่ได้เป็นพวกเดียวกันมารวมกัน ทางเลือกของการรวบรวม เมื่อฟังก์ชันผลิตค่าออกมาแทนที่จะใช้ state ที่เปลี่ยนได้ร่วมกัน คือ **Combine Functions into Transform**
