## Intent

เมื่อมี function อิสระหลายตัวที่ทำงานบนกลุ่มข้อมูลชุดเดียวกัน — แต่ละตัวรับมันเข้ามาเป็น argument แต่ละตัวคำนวณค่าจากมัน — ให้รวมพวกมันเข้าเป็นคลาส ข้อมูลที่ใช้ร่วมกันกลายเป็น field ของอ็อบเจกต์ function กลายเป็น method ของมัน และ data clump ก็เลิกถูกร้อยส่งผ่านทุก ๆ การเรียก

## The smell

คุณเห็นกลุ่ม function ที่รับ record ตัวเดียวกันและดึงค่าออกจากมัน: `baseCharge(reading)`, `taxableCharge(reading)`, `calculateBaseCharge(reading)` record เดินทางจาก function หนึ่งไปอีก function ในรูป parameter และ function เหล่านี้ก็รวมเป็นครอบครัวรอบ ๆ มันอย่างชัดเจน — แต่ไม่มีอะไรตั้งชื่อให้ครอบครัวนั้น นี่คือ **Data Clump** ที่มี behavior โคจรอยู่รอบ ๆ การห่อข้อมูลไว้ในคลาสจะให้บ้านแก่ครอบครัวนี้ ให้ method เรียกกันเองได้โดยไม่ต้องส่งข้อมูลซ้ำ และสร้างที่ชัดเจนให้ function ที่เกี่ยวข้องตัวต่อไปมาลง

## Before → After

function สามตัวที่แต่ละตัวรับ `reading` ของมิเตอร์แล้วคำนวณค่าใช้จ่าย เรารวมพวกมันเข้าเป็นคลาส `Reading` ที่มี field เป็นข้อมูลของ reading

## Mechanics

1. เลือก record ร่วมที่ทุก function ใช้ และทำ **Encapsulate Record** ถ้ามันเป็นโครงสร้างข้อมูลเปล่า ๆ — คุณต้องการอ็อบเจกต์หนึ่งตัวมายึดคลาสไว้
2. สร้างคลาส (หรือ struct + impl) ที่รับ record นั้นใน constructor และเก็บมันไว้เป็น field
3. ย้ายแต่ละ function เข้าคลาสทีละตัว เปลี่ยน parameter ข้อมูลของมันให้เป็นการอ้างถึง field ที่เก็บไว้ แล้วรันเทสต์หลังการย้ายแต่ละครั้ง
4. แทนที่การเข้าถึง parameter เดิม (เช่น `reading.month`) ด้วยการเข้าถึง field บนอ็อบเจกต์
5. เมื่อ method เริ่มเรียกกันเอง ให้ทิ้ง argument ที่ตอนนี้ซ้ำซ้อนที่เคยส่งกันออกไป
6. **รันเทสต์** หลังทุกขั้น แล้วอัปเดต call site เดิมแต่ละจุดให้สร้างอ็อบเจกต์และเรียก method

## When to use / trade-offs

ใช้สิ่งนี้เมื่อชุดของ function เป็นของข้อมูลชิ้นเดียวอย่างชัดเจน และคุณคอยส่งข้อมูลนั้นไปมาระหว่างพวกมัน คลาสให้ชื่อแก่กลุ่มก้อนนี้ ลบการร้อย parameter ที่ซ้ำซาก และเสนอบ้านตามธรรมชาติให้ตรรกะที่เกี่ยวข้องซึ่งคุณจะเพิ่มเข้ามาภายหลัง มันยังเข้าคู่ได้ดีกับ **Split Phase** และ **Extract Function** — เมื่อ function ใช้อ็อบเจกต์ร่วมกัน ค่าที่คำนวณได้ก็กลายเป็น query ที่สะอาดบนมัน

ข้อแลกเปลี่ยนคือคุณกำลังนำอ็อบเจกต์เข้ามาในที่ที่เคยมีแต่ function ธรรมดา ซึ่งคุ้มก็ต่อเมื่อ function เหล่านั้นเกาะกลุ่มกันรอบข้อมูลร่วมจริง ๆ ถ้า function เพียงบังเอิญรับ argument คล้ายกันแต่มุ่งเป้าที่ไม่เกี่ยวกัน คลาสก็จะจับสิ่งที่ไม่ได้เป็นพวกเดียวกันมารวมกัน ทางเลือกของการรวบรวม เมื่อ function ผลิตค่าออกมาแทนที่จะใช้ state ที่เปลี่ยนได้ร่วมกัน คือ **Combine Functions into Transform**

| ใช้ Combine Functions into Class เมื่อ | หลีกเลี่ยงเมื่อ |
|---|---|
| function หลายตัวใช้ data record เดียวกันเสมอ | function เหล่านั้นไม่มี shared state จริง ๆ |
| ต้องการ method ที่ computed จาก shared data | data นั้นเป็น primitive ที่ไม่ควร wrap |
| ต้องการส่ง "bundle of functions + data" ไปรอบ ๆ | เพื่อ convenience เท่านั้น ไม่ใช่ cohesion |

> ⚠️ ไม่ควร Combine Functions into Class เมื่อ:
> - function เหล่านั้นทำงานกับ data ที่ต่างกัน — cohesion ต่ำ
> - สร้าง class เพื่อ OOP แต่ภาษาที่ใช้ถนัด functional มากกว่า
> - ยังไม่แน่ใจว่า data เหล่านี้จะอยู่ด้วยกันตลอด

<Callout title="จับกลุ่มรอบข้อมูล ไม่ใช่รอบกริยา">
เหตุผลที่จะรวม function เหล่านี้คือข้อมูลที่พวกมันใช้ร่วมกัน ไม่ใช่เพราะชื่อมันคล้องจองกัน ถ้าทุกตัวรับ record เดียวกันและอ่านจากมัน คลาสก็เขียนตัวมันเอง ถ้าพวกมันแค่ดูคล้ายกันแต่แตะข้อมูลคนละชุด ก็ปล่อยให้เป็น function แยกกันไว้
</Callout>
