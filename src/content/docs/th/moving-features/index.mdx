## Intent

กลุ่ม **moving features** เป็นเรื่องของ *ตำแหน่ง* การ refactor แบบ composing-methods จะปรับรูปร่างภายในของ function แต่การย้ายเหล่านี้จะตัดสินว่า function ฟิลด์ หรือกลุ่ม behavior ทั้งก้อนควรไปอยู่ในคลาสหรือโมดูลใด โปรแกรมยังคงทำสิ่งเดิมหลังจากนั้น — แต่งานแต่ละชิ้นจะอยู่ติดกับข้อมูลและผู้ร่วมงานที่มันใช้จริง

## The smell

การตัดสินใจเรื่องซอฟต์แวร์ในวันแรกแทบไม่เคยรอดมาถึงวันที่ร้อย function ที่เคยเป็นของ `Order` ค่อย ๆ เติบโตจนอ่านข้อมูลส่วนใหญ่จาก `Customer` ฟิลด์ที่เพิ่มเข้าไปใน record หนึ่งกลับถูกแตะต้องทุกที่ยกเว้น record นั้น คลาสเล็ก ๆ ค่อย ๆ ดูดซับความรับผิดชอบที่สองเข้ามาอย่างเงียบ ๆ จนกลายเป็นสองคลาสที่สวมชื่อเดียวกัน เมื่อ behavior และข้อมูลที่มันพึ่งพิงห่างกันออกไป ทุกการเปลี่ยนแปลงก็หมายถึงการกระโดดไปมาระหว่างไฟล์ การย้ายในโมดูลนี้จะดึงมันกลับมาอยู่ด้วยกัน

| Code Smell | อาการที่บ่งชี้ | ท่าที่แนะนำ |
|---|---|---|
| Feature Envy | function อ้างอิงข้อมูลของ class อื่นมากกว่า class ตัวเอง | Move Function |
| Data Clumps | ฟิลด์ชุดเดิมเดินทางไปด้วยกันในหลาย function | Move Field |
| Large Class | class ทำสองงานที่ไม่เกี่ยวกัน เปลี่ยนด้านหนึ่งกระทบอีกด้าน | Extract Class |
| Inappropriate Intimacy | caller เดินตามสาย object เพื่อถึง dependency | Hide Delegate |
| Middle Man | class ส่งต่อทุก method call โดยไม่เพิ่มค่า | Inline Class |

## What this module covers

การ refactor เก้าแบบ ซึ่งแต่ละแบบย้อนกลับได้:

- **[Move Function](/refactoring/th/moving-features/move-function/)** — เมื่อ function อ้างอิงข้อมูลและ method ของอ็อบเจกต์อื่นมากกว่าของตัวเอง (Feature Envy) ให้ย้ายมันไปยังคลาสที่มันอิจฉา
- **[Move Field](/refactoring/th/moving-features/move-field/)** — เมื่อฟิลด์ถูกอ่านและเขียนจาก record อื่นมากกว่าบ้านปัจจุบันของมัน ให้ย้ายมันไปยังที่ที่ถูกใช้งาน
- **[Extract Class](/refactoring/th/moving-features/extract-class/)** — เมื่อคลาสหนึ่งเติบโตจนทำสองงานที่ต่างกัน (Large Class) ให้แยกความรับผิดชอบหนึ่งออกไปเป็นคลาสใหม่
- **[Inline Class](/refactoring/th/moving-features/inline-class/)** — สิ่งตรงข้าม: เมื่อคลาสไม่คุ้มกับการมีอยู่อีกต่อไป ให้พับมันกลับเข้าไปในคลาสที่ใช้มัน
- **[Hide Delegate](/refactoring/th/moving-features/hide-delegate-and-remove-middle-man/)** (และสิ่งตรงข้ามคือ **Remove Middle Man**) — ห่อหุ้มการมอบหมาย (delegation) เพื่อให้ผู้เรียกเลิกเดินไล่ตามสายของอ็อบเจกต์ ตามหลัก Law of Demeter
- **[Move Statements into Function](/refactoring/th/moving-features/move-statements-into-function/)** — เมื่อคำสั่งเดียวกันรันอยู่ข้างการเรียก function เสมอ ให้พับมันเข้าไปใน function เพื่อให้ผู้เรียกลืมไม่ได้
- **[Move Statements to Callers](/refactoring/th/moving-features/move-statements-to-callers/)** — สิ่งตรงข้าม: เมื่อ function ทำงานเกินไปหนึ่งอย่างสำหรับผู้เรียกบางราย ให้ดันคำสั่งที่แตกต่างกันออกไปยังผู้เรียกเหล่านั้น
- **[Split Loop](/refactoring/th/moving-features/split-loop/)** — เมื่อลูปหนึ่งทำสองงานที่ไม่เกี่ยวข้องกัน ให้แยกมันออกเป็นสองลูปที่ทำงานเดียวเพื่อความชัดเจน
- **[Combine Functions into Class](/refactoring/th/moving-features/combine-functions-into-class/)** — เมื่อ function คอยส่งข้อมูลก้อนเดิมไปมา ให้รวบมันเข้าเป็นคลาสที่เก็บข้อมูลนั้นไว้เป็นฟิลด์

แต่ละบทเรียนแสดง before-and-after เดียวกันใน TypeScript, Python, Go และ Rust จากนั้นพาเดินผ่านกลไก (mechanics) ที่ปลอดภัยและเป็นลำดับขั้น — โดยรันเทสต์ของคุณระหว่างทุกขั้นตอน

<Callout title="ย้าย ไม่ใช่เขียนใหม่">
ทุกการย้ายในที่นี้รักษา behavior ที่สังเกตได้ คุณกำลังย้ายตำแหน่ง code ไม่ใช่ออกแบบมันใหม่ ทำการย้ายเป็นขั้นเล็ก ๆ พร้อมชุดเทสต์ที่เป็นสีเขียว และเมื่อมันลงตัวแล้ว *เท่านั้น* คุณจึงค่อยพิจารณาเปลี่ยนสิ่งที่ code ที่ย้ายมานั้นทำ
