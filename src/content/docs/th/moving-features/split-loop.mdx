## Intent

ลูปเดียวมักถูกผลักให้ทำสองสิ่งที่ต่างกันพร้อมกัน — รวมค่าสิ่งหนึ่งขณะหาค่ามากที่สุดของอีกสิ่ง หรือเก็บรวบรวมชื่อขณะนับจำนวนที่ตรงกัน ให้แยกมันออกเป็นสองลูป แต่ละลูปวนคอลเล็กชันเดียวกันแต่ทำงานเพียงอย่างเดียวเป๊ะ ๆ ตอนนี้แต่ละลูปอ่านเป็นแนวคิดเดียว และคุณมีอิสระที่จะ refactor หรือย้ายแต่ละอันได้อย่างอิสระ

## The smell

คุณอ่านลูปหนึ่งแล้วพบว่าตัวเองกำลังสางเส้นด้ายสองเส้นในใจ: "ส่วนนี้สร้างยอดรวม… และส่วนนี้ ซึ่งอยู่ในนี้ด้วย ติดตามคนที่อายุน้อยที่สุด" ตัวสะสมสองตัวที่ไม่มีอะไรเกี่ยวข้องกันใช้ตัวลูปร่วมกันเพียงเพราะมันบังเอิญเดินไปบนลิสต์เดียวกัน การจับคู่นั้นทำให้ลูปตั้งชื่อยาก extract ยาก และเปลี่ยนยาก — แตะตรรกะยอดรวมแล้วคุณเสี่ยงกับตรรกะคนที่อายุน้อยที่สุด

## Before → After

ลูปที่วนพนักงานเพื่อคำนวณเงินเดือนรวม *และ* หาอายุที่น้อยที่สุดซึ่งไม่เกี่ยวข้องกัน สองงาน หนึ่งลูป เราแยกมันออก

## Mechanics

1. Copy ลูปทั้งหมดเพื่อให้คุณมีสองลูปที่เหมือนกันวนคอลเล็กชันเดียวกัน
2. ในลูปแรก ลบคำสั่งที่เป็นของงาน *ที่สอง* ในลูปที่สอง ลบคำสั่งที่เป็นของงาน *แรก* ตอนนี้แต่ละลูปทำงานเดียว
3. **รันเทสต์ของคุณ** พฤติกรรมต้องไม่เปลี่ยนแปลง
4. ใช้ **Extract Function** กับแต่ละลูปเพื่อให้มันกลายเป็น query ที่มีชื่อดีอย่าง `totalSalary` หรือ `youngestAge`
5. **รันเทสต์ของคุณ** อีกครั้ง — ตอนนี้ผู้เรียกอ่านเป็นผลลัพธ์ที่มีชื่อสองอย่างแทนที่จะเป็นลูปเดียวที่พันกัน

## When to use / trade-offs

แยกลูปเมื่อใดก็ตามที่ตัวมันแบกความรับผิดชอบสองอย่างที่คุณจะอธิบายด้วยคำว่า "และ" ผลตอบแทนคือความชัดเจนและการนำกลับมาใช้ใหม่: แต่ละลูปตั้งชื่อง่าย extract ง่าย และเปลี่ยนได้อย่างอิสระ — และ Split Loop มักเป็นก้าวแรกสู่การแทนที่แต่ละชิ้นด้วย pipeline operation ที่ชัดเจน

ข้อโต้แย้งที่ชัดเจนคือเรื่องประสิทธิภาพ: ตอนนี้คุณวนสองครั้ง ในโค้ดจริงเกือบทั้งหมด ค่าใช้จ่ายนั้นแทบไม่มีนัยเมื่อเทียบกับสิ่งที่ได้มาด้านความอ่านง่าย และการวนครั้งเดียวคือ premature optimisation ถ้าภายหลังการ profile พิสูจน์ว่าลูปที่ร้อนแรงนั้นสำคัญ คุณก็รวมมันกลับได้ — แต่จงปรับ optimize จากโค้ดที่สะอาดและมีชื่อดี ไม่ใช่จากความพันกัน
