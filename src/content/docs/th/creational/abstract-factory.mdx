## เจตนา

Abstract Factory ให้ interface เดียวสำหรับสร้างตระกูลของ product ที่เกี่ยวข้องกันโดยไม่ต้องเอ่ยชื่อ concrete class ของพวกมัน รับประกันว่า product ที่คุณได้มาจะอยู่ในตระกูลเดียวกันเสมอ

## ปัญหา

ลองพิจารณา UI kit ข้ามแพลตฟอร์ม หน้าต่างหนึ่งต้องการปุ่มและ checkbox และในทุกแพลตฟอร์ม widget เหล่านั้นต้องมีหน้าตาและพฤติกรรมที่สอดคล้องกัน ปุ่มแบบ macOS ที่จับคู่กับ checkbox แบบ Windows ย่อมผิดเพี้ยน อันตรายคือการปนตระกูลกัน โค้ดที่สร้างแต่ละ widget แยกกันอาจประกอบชุดที่ไม่สอดคล้องกันได้ง่าย ๆ

Abstract Factory แก้ปัญหานี้ด้วยการยกทั้งตระกูลขึ้นมาเป็น interface เดียว `GuiFactory` ประกาศ `createButton` และ `createCheckbox` แต่ละ concrete factory (`MacFactory`, `WindowsFactory`) คืน widget จากแพลตฟอร์มเดียวเท่านั้น แอปพลิเคชันรับ factory มาครั้งเดียว แล้วจึงขอชิ้นส่วนต่าง ๆ จากมัน เพราะทุกชิ้นมาจาก factory เดียวกัน จึงรับประกันได้ว่าพวกมันเข้ากันได้ และการสลับแพลตฟอร์มก็หมายถึงการสลับออบเจ็กต์เพียงตัวเดียว

**เปรียบเทียบกับ Factory Method** Factory Method ว่าด้วย product *หนึ่ง* ตัวที่เลือกผ่านการ subclass และ override method เดียว ความแปรผันอาศัยการสืบทอด ส่วน Abstract Factory ว่าด้วย *ตระกูลของ product* ที่เลือกผ่านการประกอบเข้าใน factory object โดยทั่วไปคุณถือ factory instance และเรียก creation method หลายตัวบนมัน พูดอีกอย่างคือ Factory Method ตอบคำถาม "ตัวไหน?" ผ่าน subclass ในขณะที่ Abstract Factory ตอบคำถาม "ชุดไหน?" ผ่านออบเจ็กต์ที่สลับได้ และ method ต่าง ๆ ของ Abstract Factory ก็มักเป็น factory method เสียเอง

<Mermaid code={abstractFactoryUml} title="แต่ละ concrete factory สร้างหนึ่งตระกูลของ widget ที่สอดคล้องกัน" />

- **Abstract Factory** (`GuiFactory`) — ประกาศ creation method หนึ่งตัวต่อ product หนึ่งตัวในตระกูล
- **Concrete Factories** (`MacFactory`, `WindowsFactory`) — แต่ละตัวผลิตตระกูลที่ครบถ้วนและสอดคล้องกันภายใน
- **Abstract Products** (`Button`, `Checkbox`) — interface ที่ client พึ่งพา
- **Concrete Products** (`MacButton`, `WindowsCheckbox`, …) — implementation เฉพาะแพลตฟอร์ม
- **Client** — รับ factory มาและสร้าง UI จากชิ้นส่วนที่มันคืนกลับมา โดยไม่เคยเอ่ยชื่อ concrete class

    from abc import ABC, abstractmethod


    class Button(ABC):
        @abstractmethod
        def render(self) -> str: ...


    class Checkbox(ABC):
        @abstractmethod
        def render(self) -> str: ...


    class MacButton(Button):
        def render(self) -> str:
            return "[ macOS button ]"


    class MacCheckbox(Checkbox):
        def render(self) -> str:
            return "[ macOS checkbox ]"


    class WindowsButton(Button):
        def render(self) -> str:
            return "[ Windows button ]"


    class WindowsCheckbox(Checkbox):
        def render(self) -> str:
            return "[ Windows checkbox ]"


    class GuiFactory(ABC):
        @abstractmethod
        def create_button(self) -> Button: ...

        @abstractmethod
        def create_checkbox(self) -> Checkbox: ...


    class MacFactory(GuiFactory):
        def create_button(self) -> Button:

            return MacCheckbox()


    class WindowsFactory(GuiFactory):

            return WindowsButton()

        def create_checkbox(self) -> Checkbox:
            return WindowsCheckbox()


    def build_form(factory: GuiFactory) -> str:
        return f"{factory.create_button().render()} {factory.create_checkbox().render()}"

## เมื่อไรควรใช้ / ข้อแลกเปลี่ยน

- **ข้อดี:** product จาก factory เดียวรับประกันว่าอยู่ในตระกูลเดียวกัน ไม่มีการจับคู่ที่ผิดเพี้ยน
- **ข้อดี:** การสลับทั้งตระกูลเป็นการเปลี่ยนแปลงเพียงบรรทัดเดียว: ส่ง factory ตัวอื่นเข้าไป
- **ข้อดี:** client พึ่งพาเฉพาะ abstract product interface เท่านั้น ไม่เคยพึ่งพา concrete class
- **ข้อเสีย:** การเพิ่ม *product ใหม่* เข้าตระกูลหมายถึงการต้องเปลี่ยน factory interface และทุก concrete factory
- **ข้อเสีย:** สำหรับ product เดียวหรือตระกูลเดียว มันหนักกว่า Factory Method หรือ constructor ธรรมดา

## Pattern ที่เกี่ยวข้อง

- [Factory Method](/design-patterns-from-zero-to-hero/th/creational/factory-method/) เป็นญาติที่เรียบง่ายกว่าซึ่งจัดการ product เดียว และมักถูกใช้ implement แต่ละ method ของ Abstract Factory
- [Singleton](/design-patterns-from-zero-to-hero/th/creational/singleton/) มักถือ factory ที่เลือกไว้เพื่อให้ instance เดียวถูกแชร์ทั่วทั้งแอป

<Callout title="ตระกูล ไม่ใช่สมาชิก">
หยิบ Abstract Factory มาใช้เมื่อ product มาเป็น *ชุด* ที่ต้องสอดคล้องกัน ถ้าคุณแปรผันเพียง product เดียวเสมอ Factory Method เบากว่าและชัดเจนกว่า
</Callout>
